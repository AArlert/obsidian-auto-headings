# obsidian-auto-headings

> 一款 Obsidian 插件，两层价值（定位论证见 [附录 A](#附录-a--方向审查与否决理由grill) §A.1）：
> **① 改标题不断链**——`[[file#heading]]` 内部链接随标题改写自动同步，可独立于编号工作（1.0.8 起）；
> **② 全社区最强的标题自动编号**——完全自定义多模板、按路径配置、白名单豁免，编号直接写入正文（burn-in）。
> 界面中英双语（0.6.5 起，自动跟随 Obsidian / 可手动锁定）。

## 1. 背景

Obsidian 社区呼声第一的痛点是**改标题断链**：`[[file#heading]]` 式内部链接在标题改名后即刻失效，官方无解，既有方案（Header Enhancer）还要求先启用其编号功能。本插件把「标题改名不断链」作为第一价值：拥有全社区最可靠的改名检测引擎（快照基线 + 按序配对），并自 1.0.8 起可完全独立于编号触发（CR-18；开关保守默认关，稳定后翻默认以兑现「装上即不断链」的零配置目标——定位倒转的论证见 [附录 A](#附录-a--方向审查与否决理由grill) §A.1）。

在此之上是插件的第二层价值、也是历史起点：**标题自动编号**（面向写规格、论文、法律文书等需要编号进正文的场景）。手动维护编号极易出错——中间插入一节就要重编其后所有内容；本插件挂接编辑器事件，在可配置的防抖延迟后检测标题变化，把编号前缀直接写入文件（burn-in 哲学，编号是真文本而非渲染层装饰）。每一级标题的显示格式（中文数字、法律编号、字母列表等）完全可自定义，规则可按文件夹乃至单个文件分别配置；编号引发的标题改写同样自动同步内部链接。

## 2. 需求分析

### 2.1 核心需求

| 编号 | 需求描述 |
|------|----------|
| CR-01 | 在可配置的防抖延迟（默认 300 ms）后检测标题变化，并将编号前缀写入当前活跃的 `.md` 文件 |
| CR-02 | 编号直接写入文件（而非仅注入渲染层） |
| CR-03 | 每个模板可配置**起始编号层级 `topLevel`**（默认 H2）：比它浅的标题完全不编号、且**不被改写**；它及更深的标题正常编号，并以它为序号第一段。默认 H2 即「H1 作标题/分节、不编号」 |
| CR-04 | **插件永不改写标题的 `#` 层级**（不对多个 H1 做任何降级/重排）。文件中可有任意多个 H1：低于 `topLevel` 的标题原样保留，但仍充当计数器**重置边界**（其后更深标题重新从 1 起）；需对个别标题（如某行标题）豁免编号时用白名单 |
| CR-05 | 命中白名单的标题不添加编号前缀，且**不占用计数器槽位**（计数器不累加、不跳号）。白名单支持**三种匹配方式**——**全部匹配**（修剪空格后完全相等）、**部分匹配**（标题包含该词）、**子树匹配**（命中标题及其下属所有子标题整体豁免）；当同一标题被多条目命中时按优先级 **全部 ＜ 部分 ＜ 子树** 取最强者。**例外（决策 D1，0.7.10）**：**子树**豁免块视为独立结构，块结束后计数器**整体重置**、其后编号重新开始（见 [3.7](#37-白名单系统)） |
| CR-06 | 白名单为**模板级**配置，随模板生效（路径专属模板覆盖全局默认模板）；所有配置**仅在插件设置 GUI 中管理**，不写入 frontmatter |
| CR-07 | 每个标题级别的显示格式由**模板**定义，模板完全可由用户自定义；每级可单独设置**继承前级**（默认开启，可关闭），关闭后该级不拼接父级序号，仅呈现本级序号 |
| CR-08 | 可定义多个具名模板；路径匹配规则将每个文件夹（或文件）映射到指定模板 |
| CR-09 | 插件内部使用纯阿拉伯数字计数器追踪各标题级别；显示格式（序号样式与间隔符）仅在写入文件时才应用 |
| CR-10 | 每次触发时对整个文件重新编号（而非仅修改编辑点附近的区域），确保全局一致性 |
| CR-11 | 提供**面板全局开关**与对应的 Obsidian **全局命令**，可整体启用/禁用自动编号 |
| CR-12 | 支持通过 frontmatter 键 `obsidian-auto-headings`（类型：Obsidian 复选框；值 `true`/`false`）对单个文件局部控制；非法值（含旧版 `ON`/`OFF` 文本、大小写不符者）忽略 |
| CR-13 | 界面**中英双语**（0.6.5 落地）：自动跟随 Obsidian 语言 / 可手动锁定，文案集中于 `src/i18n.ts`（见 [3.11](#311-国际化中英双语m6)） |
| CR-14 | 模板中每级标题可单独配置**继承前级**（布尔，默认开启）；关闭时该级不继承父级序号，编号前缀仅为「前缀 + 本级序号 + 标题间隔符」 |
| CR-15 | 提供侧栏大纲面板，实时展示当前文件的编号后标题树，支持按层级筛选、按文本/编号搜索并保留树形结构、点击跳转编辑器（M8a，规划中，见 [3.14](#314-侧栏大纲导航m8a)） |
| CR-16 | 支持在大纲面板中拖放调整标题的层级与顺序，操作后一次性完成重新编号与 backlink 同步；支持在面板内直接编辑标题文本（M8b，规划中，见 [3.15](#315-拖放重排与结构编辑m8b)） |
| CR-17 | 提供单文件内的目录生成能力：专属围栏代码块内自动生成、并随标题结构持续同步当前文件的目录，写入真实 Markdown 文本（而非渲染层展示）；更新联动编号引擎的防抖触发路径，且不得扰动用户当前光标/视图位置（M10，规划中，见 [3.16](#316-原生风格-toc-burn-inm10)） |
| CR-18 | Backlink 同步可独立于编号模板触发：即便某文件未命中任何编号模板（或全局自动编号关闭且该文件未 frontmatter 强制开启），只要标题文本被改写，仍应检测改名并同步引用链接，不以"编号引擎本次是否有写入"为前提（**1.0.8 落地为独立 opt-in 开关，1.0.9 并入 `updateBacklinks` 单开关全局生效**，见 [3.12](#312-backlink-同步m710-发布前置)） |

### 2.2 非目标

- 仅注入渲染层（CSS 计数器）——编号必须在原始 Markdown 中可见。**候选翻案中（2026-07-10，见
  [附录 A](#附录-a--方向审查与否决理由grill) §8.5 与 [M9 候选](#milestone-9--后续候选功能)）**：burn-in 作为**默认哲学**
  维持不变；但「虚拟编号模式」（显式 opt-in：编号仅作 CM6 widget + 阅读视图 postprocessor 渲染、
  永不写文件）已立为候选——面向"生态纯净优先"人群的第二哲学，该模式下零 WJ、编号引发的断链问题
  整体消失、阅读模式天然支持。翻案逻辑同 TOC 先例：与核心哲学的关系想清楚了（并列而非替代）
- **改写用户的标题层级**（插件只增删编号前缀，绝不把 `#` 改成 `##`；多个 H1 的"规整"交给用户/白名单）
- ~~生成目录~~ **决策已翻案（0.7.28，见 [3.16](#316-原生风格-toc-burn-inm10)）**：单文件内的目录不再是
  非目标——burn-in 真实 Markdown 文本这件事，Dataview（渲染层，依赖额外插件才能看）与 Table of
  Contents 插件（一次性插入静态文本，标题改了不会跟着更新）都做不到，与本插件"写入真文本、不做渲染
  覆盖层"的核心哲学一致，理由与 Backlink 同步（同样是"别的插件做不到"）同源。**跨文件聚合视图**（总
  索引、跨笔记聚合）仍是非目标，继续走 Roadmap M9「Dataview 集成」路线——单文件目录与跨文件聚合视图
  是两个不同范围的问题，互不替代
- 在 frontmatter 中存储任何**配置**（模板、白名单、路径规则刻意排除，以保持笔记整洁）；frontmatter 仅用作单文件开关
- 多文件批量重新编号（后续版本）
- Git 感知最小化差异（后续版本）
- 标题居中 / 自定义字体等**渲染层美化**（与「burn-in 写入」哲学冲突，明确不做；有此需求可用 Auto Heading）

### 2.3 边界情况与设计决策

> 本表是**规格层**的边界决策；逐条**可操作 / 可断言**的测试场景（含"改配置后再触发"这类状态转移、
> 当前已知 bug 与其状态）见 [`doc/testplan.md`](./testplan.md)。改规格时回头核对 testplan 是否需同步。

| 场景 | 决策 |
|------|------|
| 文件中无 H2 及以上标题 | 插件不写入任何内容，文件保持不变 |
| 白名单标题有子标题 | **全部/部分匹配**：仅该标题本身跳过前缀、不占计数器槽位；其子标题会继续沿用**上一个已编号祖先**的计数（通常非预期，面板会 ⚠ 提示改用子树）。**子树匹配**：该标题及其下属所有子标题整体跳过前缀、均不占槽位 |
| 用户手动编辑了编号前缀 | 下一个防抖周期会将其覆盖为正确值；对前缀的手动编辑是预期行为。**修订预告（2026-07-10）**：[M11 的 CM6 原子区域](#milestone-11--信任包插队-m8a-之前)落地后此原则退役——编辑器内前缀只读（光标跳过、不可落点），编辑器外损伤仍由双哨兵自愈兜底（见 [附录 A](#附录-a--方向审查与否决理由grill) §8.4） |
| 标题正文本身以数字起头（如手写 `## 2024 总结`） | **方案A（0.6.6）根治**：`stripPrefix` 只认带 Word Joiner 的（插件自己写的）前缀，`## 2024 总结` 无 WJ → `2024 总结` 整体是正文 → 首次即得 `## 1 ⁠2024 总结`，`2024` 完整保留、幂等（见 [2.4](#24-保留正文起头的数字方案a)）。代价：手写的伪编号（`## 1. X`）不再被自动吸收，可用「清理非本插件的标题编号」命令主动清掉（见 [3.10](#310-清除编号)）|
| 在已编号文件上改模板的**前缀 / 后缀**字段后再触发 | 不叠加（**方案 A**，0.3.18）：剥离时前后缀接受「当前值 + 空串 + 全模板在用值并集」作候选，故「无前缀时编的号」与「旧前缀值」都能被剥净，再叠正确新前缀。候选只在其后紧跟序号 token 时才构成前缀，故 `第一印象`/`API 设计` 等不受误伤（见 testplan B2/B3）|
| 在**空行**上用快捷键直接转成标题（行内无文字） | 正常编号：空行转标题后形如 `### `，编号得 `### 1.1 `（标题文本为空，末尾即标题间隔符空格）。剥离器按**保留行尾空白**的原始文本匹配，故反复触发保持幂等、不会叠成 `1.1 1.1`（早期一处剥离 bug，已修复） |
| 标题间隔符填空格 / `. ` 等**含尾随空白**的值 | **所见即所得**：用户在间隔符输入框敲什么就用什么，写入文件与序列化都**原样保留**（`" "`→`1 标题`、`". "`→`1. 标题`）。GUI 实时预览也**如实显示尾随空格**（早期预览会 trim 末尾空白，把 `" "`/`". "` 显示成 `1标题`/`1.标题`，让人误以为空格没生效——已修复，仅曾失真于预览，实际编号一直正确） |
| 存在多个 H1 | **插件不改写任何 `#` 层级**。默认 `topLevel=H2` 时所有 H1 都不编号、原样保留，但每个 H1 仍是计数器**重置边界**——其后的 H2 重新从 1 起（多个 H1 = 多个顶层分节）。想给 H1 编号则把模板 `topLevel` 设为 H1（H1 → `1`/`2`/`3`）；想让某个 H1 作"标题"不编号则用白名单豁免（见 [3.4](#34-起始编号层级与多个-h1)） |
| 标题层级跳跃（如 H3 → H5） | 缺失的中间层级如何呈现**由各模板自行配置**（占位策略 `skipFill`，见 [3.6](#36-模板系统)）：**补位**时保留一段并填入用户指定的占位字符（默认 `0` → `1.1.0.1`，也可填 `1` → `1.1.1.1` 或任意字符），使序号段数等于标题深度（H_n 有 n−1 段）；**不补位**时省略该段（H5 呈现为三段、与 H4 同形 `1.1.1`）；**不编号**（0.7.15）时跳级标题完全不编号、保持原样——面向「H5/H6 当样式性小标题」的用法，正常嵌套时照编（按上下文判定，testplan F7–F9）。无论哪种策略，缺失级计数器都不实例化（仍为 0、不被借号），随后首个真实的该级标题仍从 1 起算 |
| 白名单条目含首尾空格 / 行内格式 / 全角半角 / 大小写 | 比较前对标题与条目做同一套**归一化**（剥前缀 → 去行内 Markdown → NFKC → trim 并折叠空白 → 转小写），仅用于命中判定、不改写文件（见 [3.7](#37-白名单系统)） |
| 同一标题被多个白名单条目命中 | 取**豁免范围最大**者的并集：被任一条目命中即豁免；命中它的条目中只要有**子树**条目，则连同子树一并豁免 |
| 当前文件匹配多条路径规则 | 最具体的规则（文件路径 > 文件夹路径 > 全局默认）优先；并列时列表中靠后的规则胜出 |
| 无任何路径规则匹配当前文件 | 使用全局默认模板（`default.json`，面板显示为"默认"） |
| frontmatter 中 `obsidian-auto-headings` 值非 `true`/`false` | 视为非法，忽略该键，按缺省（跟随「全局自动编号」开关）处理；旧版 `ON`/`OFF` 文本及 `True`/`False` 大小写不符者均属非法（见 [3.2](#32-单文件开关frontmatter)） |
| 「全局自动编号」关、但某文件 `frontmatter: true` | 该文件**仍自动编号**（文件级强制 opt-in）——`true` 覆盖全局开关。用于「全局沉默、个别文档常驻自动」（见 [3.2](#32-单文件开关frontmatter)） |
| 手动「立即重新编号」命令遇到 `frontmatter: false` 或全局开关关 | **照常编号**——手动命令走独立路径，不看全局开关、不被 `false` 拦截（用户敲命令即明确意图），仅受「能否命中模板」约束（见 [3.1](#31-开关命令与生效判定)） |
| 当前文件无任何路径规则命中（含 `/` 根规则被删） | **无可用模板**：自动触发静默跳过（已有编号冻结不动）；手动命令弹 Notice「未匹配任何路径规则」。`frontmatter: true` 也不能凭空变出模板，同样静默跳过（见 [3.8](#38-按路径配置)） |
| 删除一个**正被路径规则引用**的模板 | 弹确认对话框，列出受影响规则并让用户选择去向：降级到「默认」（缺省）/ 改投他模板 / 连规则一并删（见 [3.6](#36-模板系统)） |
| 删模板 / 删路径规则后，已编号文件何去何从 | **不回滚**——配置层改动绝不批量扫库改文件；已写入的编号**冻结现状**，直到该文件下次被触发重排或被「清除编号」处理（见 [3.6](#36-模板系统)、[3.10](#310-清除编号)） |
| 想让某文件回到「无编号」裸标题 | 删模板 / 取消勾选（`false`）都不会清除已有编号；须主动用「清除当前文件编号」命令（或面板 [清除全库编号]），其剥离器用全样式并集、独立于模板（见 [3.10](#310-清除编号)） |
| 代码块内的 `#` 行 | 不视为标题，跳过处理 |
| 开启 Obsidian「页内标题」（外观 → 页内标题，将文件名作为可编辑标题显示在正文中） | **无冲突**：页内标题是由文件名生成的**渲染层 widget**，不在 Markdown 源文本里；插件只读写源文本（`editor.getValue` + 行事务），既不会给它编号也不会因它错位。此时标题改由文件名承担，若想给正文 H1 章节编号，把模板 `topLevel` 设为 H1 即可（见 [3.4](#34-起始编号层级与多个-h1)） |

### 2.4 保留正文起头的数字（方案A）

> **状态：0.6.6 已落地（方案A，下方分析末尾「结论」即最终决定）。** 本节保留完整的方案评估以备查。

**现象（修复前）：** 全局自动编号开，用户在裸标题 `## 2024 年度总结` 上触发 → 得 `## 1 年度总结`，正文里的 `2024`
被吃掉。根因同 [2.3](#23-边界情况与设计决策) 表中「标题正文以数字起头」一行：**无状态引擎分不清**「插件写的
`1 `」与「用户写的 `2024 `」——两者形状都是「数字 + 间隔符 + 其余」，`stripPrefix` **首次**就把 `2024 ` 当历史
前缀剥了，再叠上新的 `1 `。（注意：若用户在序号后把数字补回 `## 1 2024 年度总结`，0.3.18「折中」+ 0.6.4 WJ
已能**稳定保留** `2024`——本待决问题特指**首次**那一下。）

**能否让首次也不吃 `2024`？方案评估：**

| 方案 | 思路 | 评估 |
|------|------|------|
| A. **仅剥 WJ 结尾的前缀** | 0.6.4 起插件写出的前缀末尾都带 Word Joiner（U+2060）；用户手写的 `2024 ` 没有。令 `stripPrefix` **只**剥「以 WJ 收尾」的前缀，则任何正文起头数字都天然保留 | **最干净、最对症**，但有代价：① **旧文件兼容**——0.6.4 之前写入的编号无 WJ，将再也剥不掉、改触发会叠加；② B2/B3「改间隔符/前后缀后再触发」依赖容差剥历史前缀，纯 WJ 模式会回退这些能力。故不宜全局默认开 |
| B. **数字启发式**（如 ≥4 位 / 像年份的不当前缀） | 用阈值/正则猜「这是年份不是序号」 | **脆弱、locale 相关**、易误判（`12 章`？`100 题`？），否决 |
| C. **每标题手动豁免**（白名单 / frontmatter） | 把该标题加白名单 | 可行但**太手动**，且白名单是「整行不编号」，与「既编号又保留 2024」诉求不符 |
| D. **per-模板开关「保留正文起头数字」（推荐方向）** | 把方案 A 做成**模板级 opt-in**：开启后该模板的 `stripPrefix` 仅剥 WJ 结尾前缀、绝不剥无 WJ 的起头数字 | **推荐**。代价（旧前缀/换间隔符不再被容差剥）被限定在「用户主动为这个模板选择保留数字」的范围内，且 WJ 自 0.6.4 已写入，时间越久存量无 WJ 前缀越少。默认关 = 完全保持现有行为，不影响任何既有用户 |

**结论（0.6.6 已实现，方向升级）：** 最终采用比方案 D 更彻底的**方案 A 作为默认**（无线上用户、不必适配历史）：
`stripPrefix` 改为**纯 Word Joiner 边界**——**只剥插件自己写的（前缀末尾带 WJ）编号，不含 WJ 的标题一律视为
纯用户文本、原样保留**。于是 `## 2024 年度总结`（无 WJ）首次触发即得 `## 1 ⁠2024 年度总结`，`2024` 完整保留、
反复触发幂等。这同时**根治了 U1/U2/U3**（见 testplan §3.2）——那一整类「正文被容差正则当编号吃掉」的问题，
本质都是「分不清插件前缀与用户文本」，WJ 边界把它一次性消除。

**配套取舍（已接受）：**
- 0.6.4 之前（无 WJ）写入的历史前缀不再被常规重排识别——无线上用户，不适配。
- 用户**手写**的伪编号（`## 1. 标题`、`## 第三章 X`、导入文档里的 `1.2 X`）不再被自动「吸收」，而是叠加在其上。
  为此新增手动命令 **「清理非本插件的标题编号」**（见 [3.10](#310-清除编号)）：一键剥掉这类手写 / 外来编号，
  好让插件接管。
- 「清除编号」命令（{@link stripPrefixBroad}，全样式正则、独立于 WJ）仍能清掉手写 / 历史前缀，不受影响。

### 2.5 双哨兵自愈：删后缀致序号重复的根治（0.7.20）

> **参考实现：** 用不可见 Word Joiner 字符标记编号前缀边界这一思路，最初参考自
> gurjar1/auto-heading-obsidian（README 原文："tagged with an invisible Unicode marker (U+2060)
> so they can be identified with certainty"）；本节的「首尾双哨兵」是在此基础上的进一步升级
> （单哨兵在后缀被部分删除时无法自愈，见下）。见「关于」TAB 鸣谢。

**问题（用户报告）。** 标题 `## 一、⁠标题2`（`一` 序号 + `、` 后缀 + 不可见 WJ + 正文），用户「从 `、`
开始删」时，把 `、` 连同紧随其后的**不可见 WJ 一起删掉**，行变成 `## 一标题2`（无 WJ）。下一个防抖周期，
方案 A 的「无 WJ 一律正文」把 `一标题2` 整段当正文，左侧再叠一层新前缀，得到 `## 一、⁠一标题2`——**序号
`一` 重复**。根因是方案 A 的固有张力：WJ 是唯一边界证据，一旦被编辑破坏，残留的序号字与真实正文（如
`## 2024 总结` 的 `2024`）**无从区分**——保守（不剥）则重复，激进（剥）则重蹈 E5 误伤。

**方案对比。** 见 [testplan E14–E18](testplan.md) 与本次实现记录：

| 方案 | 做法 | 结论 |
|------|------|------|
| ① 精确去重 | 正文恰以「将写入前缀」开头时剥一处 | **弃用**：双哨兵均毁时无证据，会把 `## 1。2024总结` 的正文 `1。` 当孤儿吃掉（回归 E5/U2） |
| ② **双哨兵**（采用） | 前缀**首尾各带一个 WJ 哨兵**（`⁠一、⁠标题2`） | **采用**：见下 |
| CodeMirror 原子区域 | 视图层把前缀设为不可编辑 | **已升格（2026-07-10，系统论证见 [附录 A](#附录-a--方向审查与否决理由grill) §8）**：不能**替代** WJ（装饰是视图态、答不了跨会话/跨设备的「身份」问题，也护不住编辑器外的写入），但作为 WJ **之上**的第四层防护 + 体验层立项进 [M11 信任包](#milestone-11--信任包插队-m8a-之前)——防护栈变为：原子区域（编辑器内让破坏根本不发生）→ 方案A → 双哨兵自愈 → 清除命令兜底；「前缀可手改」原则届时退役（见 [2.3](#23-边界情况与设计决策) 对应行修订预告） |

**② 双哨兵机制。** `buildPrefix` 写出 `WJ + 前缀内容 + WJ`。尾哨兵（贴正文、最易在改后缀时被误删）被删
时，**首哨兵**（在行首 `#{1,6} ` 之后、远离用户编辑后缀的位置）幸存，成为「此处确有插件前缀」的**结构性
证据**。`stripPrefix` 据此在**有证据**的前提下启用有界剥离（`boundedStripDamagedPrefix`，按模板样式消费残缺
前缀），把孤儿序号剥净并重叠正确前缀 → **愈合**回 `## 一、⁠标题2`。这把方案 A 禁止的「宽松剥离」在证据约束下
安全地重新引回，不误伤无证据的正文（`2024` 从无首哨兵 → 永不触碰）。

**③ 降级残留清理。** 用户把 `## 标题` 的 `#` 删光降级为正文时，行里残留 `⁠一、⁠标题2`（WJ 哨兵 + 编号）。
`renumberContent` 与「清除编号」命令共享 `cleanDemotedResidue`：非围栏、非标题、且**以 WJ 哨兵起头**的正文行
即插件残留，剥净其前缀（WJ 是插件独有标记，正文几乎不可能自带，误伤面极小）。

**配套取舍（已接受）：**
- **首尾哨兵均被毁**（要连行首附近的首哨兵也删掉，很罕见）不自愈——与真实用户文本无从区分，强行猜测会重蹈
  E5 误伤；保留现状，由「清除编号」命令兜底。
- 每个标题多一个不可见 WJ（首哨兵）。backlink 锚点归一两侧同口径剥 WJ（`linkAnchor` 剥所有 WJ 做匹配、
  `displayAnchor` 保留 WJ 做写入），故链接不受影响；升级后首次触发会把存量标题补写首哨兵（一次性）。
- 旧**单**哨兵（0.6.4–0.7.19，WJ 在前缀末尾非首位）仍被 `stripPrefix` 正确识别（向后兼容）。

## 3. 功能规格

### 3.1 开关、命令与生效判定

插件的「是否运行」分**两层**，互不等同：

- **启用插件**（Obsidian 社区插件总开关）：决定插件代码是否加载。**启用 ≠ 自动编号。**
- **「全局自动编号」面板开关**：决定插件是否在编辑后**自动触发**编号。状态持久化于 `data.json`，与全局命令「切换全局自动编号」**双向同步**。

> **设计动机：** 很多用户只想给少数正式文档编号，不愿插件在每次打字时扰动全库。故把「装了插件」与「自动给所有文件编号」**解耦**——插件常驻，自动编号按需。关掉「全局自动编号」后，插件不再随编辑自动触碰任何文件，但用户仍可对个别文件用 `frontmatter: true` 常驻开启，或随时用「立即重新编号」命令手动触发。

**命令与按钮清单：**

| 入口 | 作用 | 触发逻辑 |
|------|------|----------|
| 命令「切换全局自动编号（全局）」 | 翻转「全局自动编号」开关，与面板双向同步 | — |
| 命令「立即重新编号（当前文件）」 | 绕过防抖，立即对当前文件执行一次完整重排 | **手动路径**（见下） |
| 命令「清除当前文件编号」 | 剥离当前文件所有本插件写入的编号前缀，不再写新前缀 | 见 [3.10](#310-清除编号) |
| 面板按钮 **[清除全库编号]** | 一次性剥离全库所有文件的编号前缀 | 手动点击 + 二次确认；刻意**不做成命令**以防误触发，见 [3.10](#310-清除编号) |

**两条生效路径**——「自动触发」与「手动命令」的判定**不同**：

```
── 自动触发（编辑后防抖）──────────────────────────────────────
「全局自动编号」= 开  或  frontmatter = true
        ↓（否则不触发）
frontmatter ≠ false
        ↓（否则不触发）
路径规则解析 → 命中模板？
        ↓（否则静默跳过，绝不打扰）
执行编号

── 手动命令（立即重新编号）────────────────────────────────────
（不看「全局自动编号」开关，也不被 frontmatter false 拦截）
路径规则解析 → 命中模板？
        ↓（否则弹 Notice：当前文件未匹配任何路径规则，无法编号）
执行编号
```

> **为何手动命令绕过开关与 `false`：** 用户主动敲命令，意图已经明确就是「现在给我编号」，不该再逼他去改 frontmatter 或翻全局开关。**手动即「我要」**，故只受「能否找到模板」约束；找不到模板时给一条 Notice 反馈（而非静默），以免用户以为命令没响应。

### 3.2 单文件开关（frontmatter）

插件读取——且**仅读取**——一个 frontmatter 键 `obsidian-auto-headings`，用于**按文件覆盖**「全局自动编号」开关。属性类型为 **Obsidian 复选框（checkbox）**，插件在加载时通过内部 API 注册，用户在属性面板点击勾选框即可操作：

```yaml
---
obsidian-auto-headings: true
---
```

- 合法值仅 `true` 与 `false`（YAML 布尔值，对应复选框勾选 / 取消勾选）。
- **`true`（勾选，文件级强制开启）**：该文件**强制参与自动编号**——即便「全局自动编号」为关，此文件仍随编辑自动重排。用于「全局沉默、个别文档常驻自动编号」的工作流。
- **`false`（未勾选，文件级强制关闭）**：该文件**不参与自动编号**——即便「全局自动编号」为开，也不自动碰它。
- **该键缺省**：跟随「全局自动编号」开关。
- **非法值**（非 `true`/`false`，含旧版 `ON`/`OFF` 文本、`True`/`False` 大小写不符者）：忽略该键，按缺省处理（跟随全局开关），不报错、不改写、不弹提示。

**仅作用于自动触发**的完整矩阵（手动命令不受此表约束，见 [3.1](#31-开关命令与生效判定)）：

| 「全局自动编号」 | frontmatter | 自动触发 |
|---|---|---|
| 开 | 缺省 / `true` | ✅ |
| 开 | `false` | ❌ |
| 关 | `true` | ✅（文件级强制） |
| 关 | 缺省 / `false` | ❌ |

- **`false` 只是「别自动碰我」，并不清除已有编号**——已写入的前缀**冻结现状**。想抹掉编号请用「清除当前文件编号」命令（见 [3.10](#310-清除编号)）；常见组合是「先清除、再取消勾选」防止下次被重编。
- 这是插件对 frontmatter 的**唯一**交互；插件绝不向 frontmatter 写入任何配置（模板、白名单、路径规则）。

### 3.3 写入文件模型

插件使用 Obsidian 的 `Editor` API 读取并改写标题行。每次触发周期执行以下步骤：

1. 逐行解析完整文件，识别标题行（以一个或多个 `#` 加空格开头且不在代码块内的行）。
2. 对**编号范围内**（`>= topLevel`）的标题剥离已有的自动编号前缀（更浅的标题不剥离，避免误伤）。
3. 基于当前模板和计数器状态为每个标题重新计算正确的前缀（**不改写任何 `#` 层级**）。
4. 以**单一事务**将所有修改写回编辑器（避免产生过多撤销记录）。

> **为何整文件重写？** 局部重写（仅修改编辑点附近）需要追踪编辑点以下哪些标题受到影响、计数器是否需要重置。整文件重写逻辑更简单、结果更正确，对于任何现实长度的笔记来说性能均可接受。

### 3.4 起始编号层级与多个 H1

**设计原则：插件只增删编号前缀，绝不改写用户的标题 `#` 层级。** 历史版本会把多余的 H1
降级成 H2（甚至级联下移子树），这等于替用户重排文档结构，自由度不足；现已彻底移除。

取而代之的是每个模板的 **起始编号层级 `topLevel`**（默认 H2，可在模板编辑面板用下拉选择 H1–H6）：

- 比 `topLevel` **浅**的标题：**完全不编号、也不被改写**，原样保留；但仍推进计数器，因此充当
  **重置边界**——其后更深的标题会重新从 1 起。
- `topLevel` **及更深**的标题：正常编号，并以 `topLevel` 为序号第一段。

**结束编号层级 `bottomLevel`（M6，默认 H6 = 无下界）：** 与 `topLevel` 对称，定义编号区间的**下界**。
比 `bottomLevel` **更深**的标题**不编号、也不被改写**（仅当其上残留旧编号时被剥除），但同样仍推进
计数器、充当重置边界。两者配合即可**只编号一个区间**，如 `topLevel=H2` + `bottomLevel=H4` 表示
「只给 H2–H4 编号，H5/H6 保持裸标题」。GUI 下拉强制 `bottomLevel ≥ topLevel`（抬高起始层级越过结束
层级时自动一并抬高结束层级），从根上避免配出空区间。引擎层 `normalizeBottomLevel` 仅把值夹到 [1,6]、
不强制大小关系，空区间（bottom < top）退化为「无任何层级编号」，无害。

**起始编号数字 `startIndex`（M8 批次 1，默认 1）：** 模板级非负整数，**仅首段**（`topLevel` 对应段）
从该值起——设 0 得 `0 / 0.1 / 1`（`0.1.1` 形态），更深层级仍从 1 起。实现为**渲染期偏移**
（`buildPrefix` 对首段加 `startIndex-1`）：计数器内部恒为 1 起，计数 0 仍是「跳级缺失」哨兵
（占位符不受偏移影响），剥离靠 WJ 边界、与序号字形无关，幂等性不受影响。
`normalizeStartIndex` 夹到 [0, 9999]，非数字 / 缺失（旧模板）回退 1。GUI 为**下拉，仅列 0 / 1**
（0.7.14 收窄自 0–9：几乎全部真实诉求是「0 起」或「1 起」，其余值等有需求再放开；JSON 手改的其他值
仍作为额外选项列出、不被静默改写）。场景真值见 testplan N 类；UVM 已纳入随机化（`setStartIndex`
激励含 2/5 等非 GUI 值，持续压引擎全域，0.7.13）。

这样各种"多个 H1"诉求都能自然满足，且插件不替用户做主：

| 诉求 | 配置 |
|------|------|
| 默认：`#` 作标题/分节、不编号，`##` 起编号 | `topLevel=H2`（多个 H1 全部原样保留） |
| 多个 H1 全部编号（无文档标题） | `topLevel=H1` → H1 得 `1`/`2`/`3`，各自 `##` 子节重置 |
| 首个 H1 作标题不编号、其余 H1 要编号 | `topLevel=H1` + 用白名单豁免那一行标题（见 [3.7](#37-白名单系统)） |

**示例**（默认 `topLevel=H2`，插件不动任何 `#`）：

```
# 第一篇            →  # 第一篇              （H1 < topLevel，不编号、原样）
## 概述             →  ## 1 概述
## 细节             →  ## 2 细节
# 第二篇            →  # 第二篇              （H1 是重置边界）
## 总结             →  ## 1 总结             ← 重新从 1 起
```

**示例**（同一文档，`topLevel=H1`）：

```
# 第一篇            →  # 1 第一篇
## 概述             →  ## 1.1 概述
# 第二篇            →  # 2 第二篇
## 总结             →  ## 2.1 总结
```

**与 Obsidian「页内标题」的关系（外观 → 页内标题）。** 该官方功能把**文件名**渲染成一个可编辑的
标题显示在正文顶部。它是编辑器从文件名生成的**渲染层元素，并不是文档里的一行 Markdown 文本**——
插件全程只读写 Markdown 源文本（`editor.getValue()` + 按行事务写回），因此**与之无任何冲突**：插件
不会给页内标题编号、也不会因它产生行错位；阅读模式下插件根本不运行。两者唯一的关系是**写作习惯的
搭配**：

| 习惯 | 推荐 `topLevel` |
|------|-----------------|
| 页内标题**关**（默认）：正文里手写 `# 文档标题`，`##` 起才是章节 | `H2`（默认，H1 不编号） |
| 页内标题**开**：文件名即标题，正文 `#` 直接是章节、希望被编号 | `H1`（正文 H1 编号为 `1`/`2`/…，文件名标题不受影响） |

> 插件**刻意不**根据「页内标题」开关自动改 `topLevel`——这与「插件绝不替用户改 `#` 层级、不替用户
> 做主」的核心原则一致（见本节开头）。是否给正文 H1 编号由你通过模板的起始编号层级显式决定。

### 3.5 内部计数模型

插件维护一个计数器数组 `[c1, c2, c3, c4, c5, c6]`，分别对应 H1–H6。序号前缀只对
`>= topLevel` 的标题输出，并从 `c[topLevel]` 起截取拼接（如默认 `topLevel=H2` 时即 `[c2..c6]`）。

规则：
- 遇到第 *n* 级标题时，`c[n]` 加一，所有 *> n* 级的计数器归零——**即便该标题低于 `topLevel`**
  （它不输出序号，但仍归零更深级别，故是重置边界）。
- 计数器数组与显示格式（序号样式、间隔符）**相互独立**；格式转换仅在写入时进行。
- 白名单中的标题**不占用计数器槽位**——计数器既不累加也不归零，因此**不产生跳号**，且不向文件写入前缀。**例外（决策 D1，0.7.10）**：**子树**豁免块结束后计数器**整体重置**（独立结构语义，见 [3.7](#37-白名单系统)）。**注意**：若被「全部/部分」豁免的标题下还有子标题，这些子标题会沿用上一个已编号祖先的计数（通常非预期，应改用「子树」豁免整块，见 [3.7](#37-白名单系统)）。
- 跨级编号（如 `1.1.1`）由各父级计数器值用模板的**序号间隔符**拼接而成；**祖先段的序号样式由模板的 `ancestorNumeral` 决定**——默认 `self`（每级父级各自套用其所在级别的样式，得 `1.a.①`），或 `arabic`（祖先一律阿拉伯，得 `1.1`，适合中文书），见 [3.6](#36-模板系统)。**若某级关闭「继承前级」，则该级不拼接父级序号，仅以本级序号呈现**（如 H5 输出 `a)`、`b)`）。
- 内部计算全程使用纯阿拉伯整数，与模板指定的序号样式无关。
- **旧前缀剥离（方案A，0.6.6）**：重排前先剥掉上一次写入的前缀，再叠新前缀。`stripPrefix` 采**纯 Word Joiner 边界**——插件写出的每个前缀末尾都带一个不可见的 WJ（U+2060），剥离时只认它：**含 WJ → 精确剥到标记之后；不含 WJ → 整段视为正文、不剥**。这样无论序号样式 / 间隔符 / 前后缀怎么改，自己写的旧前缀都能被干净剥净（WJ 与这些字段无关），而用户正文里像编号的文字（`2024 总结`）绝不会被误吃。手写 / 历史（无 WJ）前缀由专门的「清理非本插件编号」/「清除编号」命令处理（见 [3.10](#310-清除编号)）。

**示例——内部计数器状态 vs. 写入文件内容**（默认模板：序号样式=阿拉伯，序号间隔符=`.`，标题间隔符=空格）：

```
标题序列              计数器状态            写入文件（模板应用后）
──────────────       ──────────────       ────────────────────────
H2                →  c2=1            →    1 标题
H3                →  c3=1            →    1.1 标题
H3                →  c3=2            →    1.2 标题
H2                →  c2=2，c3..=0    →    2 标题
H3（白名单）       →  c3 不变（=0）   →    （不写前缀，不占槽位）
H3                →  c3=1            →    2.1 标题   ← 未跳号
```

> **跨级拼接**：H3 的编号 `2.1` 由当前 `c2`（=2）与 `c3`（=1）以序号间隔符 `.` 拼接而来。**内部计数器始终是纯阿拉伯整数**，但拼接到前缀时**每一级父级各自套用其级别的序号样式**：若 H2 用中文、H3 用阿拉伯，则 H2 显示 `一`、`二`，而 H3 的前缀里父级 H2 也呈现为中文，得 `一.1`、`二.1`（而非把父级强制成阿拉伯的 `1.1`、`2.1`）。这样把 H3 设为字母、H4 设为带圈时，H4 能呈现为 `一.a.①`，父级样式向下可见。

### 3.6 模板系统

**模板**是一个具名配置对象，为 H1–H6 各级定义编号的显示格式与该模板的白名单。模板的创建、编辑、删除**完全在插件设置 GUI 中进行**，不涉及任何 YAML 或 frontmatter。

每一级的格式由六个**结构化字段**组成（GUI 列序：前缀 → 序号 → 序号间隔符 → 后缀 → 标题间隔符 → 继承前级）：

| 字段 | 含义 | 示例 |
|------|------|------|
| 前缀（`prefix`） | 序号前的自定义文本，可为空 | `第`、`（`、空 |
| 序号样式（`numeral`） | 本级计数器的呈现形式，下拉枚举 | `arabic`(1)、`cjk`(一)、`circled`(①)、`lower-alpha`(a)、`upper-alpha`(A) |
| 序号间隔符（`numberSeparator`） | 拼接各级父子序号的符号，可完全自定义 | `.`（得 `1.1`）、`-`（得 `1-1`） |
| 后缀（`suffix`） | **完整序号之后、标题间隔符之前**的自定义文本，可为空；与前缀配合实现「第1章」式编号。作用于本级完整序号（含继承的父级序号），即 `第1.1章` 而非每段都带后缀 | `章`、`节`、`）`、空 |
| 标题间隔符（`titleSeparator`） | 完整序号与标题文本之间的文本，由用户自行输入（含可能的尾随 `.` 或空格） | `、`、` `（空格）、`. `（点+空格） |
| 继承前级（`inherit`） | 是否拼接父级序号，布尔，**默认开启**；关闭后该级仅呈现本级序号，不拼父级 | `true`（得 `1.1.1`）、`false`（H5 得 `a)`、`b)`） |

此外，每个模板还有若干**模板级**字段（不分级，作用于整模板）：

| 字段 | 含义 | 示例 |
|------|------|------|
| 起始编号层级（`topLevel`） | 从哪一级标题开始编号（下拉 H1–H6，默认 H2）：比它浅的标题不编号、不被改写（但仍是计数器重置边界），它及更深的标题正常编号并以它为序号第一段（见 [3.4](#34-起始编号层级与多个-h1)）。面板中区间外的行会置灰 | `2`（默认，H1 不编号）、`1`（H1 也编号） |
| 结束编号层级（`bottomLevel`） | 编号到哪一级为止（下拉 H1–H6，默认 H6 = 无下界；下拉仅列 ≥ 起始层级的项）：比它更深的标题不编号、不被改写（残留旧编号会被剥除），仍作重置边界。与起始层级配合可只编号 H2–H4 这样的区间（M6，见 [3.4](#34-起始编号层级与多个-h1)） | `6`（默认，无下界）、`4`（只编到 H4） |
| 跳级缺失层级（`skipFill`） | 标题跳级（如 H3 后直接跟 H5）时的策略：**补位**（`{mode:"fill",placeholder}`）保留一段并填入占位字符；**不补位**（`{mode:"drop"}`）省略该段；**不编号**（`{mode:"none"}`，0.7.15）跳级标题完全不编号、保持原样（仅剥旧前缀，仍作重置边界；正常嵌套照编——按上下文判定，testplan F7–F9）。**由各模板自行决定**，默认补 `0`。**占位字符仅限数字**（`0`→`1.1.0.1`、`1`→`1.1.1.1`）——限数字是为保证编号始终能被干净剥离、不重复叠加（数字恒被 arabic 剥离 token 识别） | `{mode:"fill",placeholder:"0"}`（默认）、`{mode:"drop"}`、`{mode:"none"}` |
| 祖先序号渲染（`ancestorNumeral`） | 继承前级时，前缀里**祖先段**（比当前级浅的各段）用什么样式呈现。**`self`（默认）**：每个祖先套用**自身级别**的序号样式（提纲惯例，H3=字母/H4=带圈 → `1.a.①`）。**`arabic`**：祖先**一律阿拉伯**、仅当前级（末段）套用自身样式（中文书惯例，H2=中文 → 标题行 `一`、H3 子节 `1.1`、`1.1.1`）。**仅影响祖先段**，末段始终套本级样式 | `self`（默认）、`arabic` |

> 「默认」模板用于所有未被路径规则单独指派模板的文件（路径规则见 Roadmap），故其 `topLevel` / `bottomLevel` / `skipFill` / `ancestorNumeral` 等价于全局默认；其他模板再各自覆盖。

> **为何需要 `ancestorNumeral`（设计取舍）：** 单个 `numeral` 字段被迫同时承担两件**方向相反**的事——「本级**单独**长什么样」与「本级**作为祖先**嵌进更深前缀里长什么样」。中文书（章 `一` / 节 `1.1`）要求祖先转阿拉伯，提纲（`1.a.①`、`I.A.1`）要求祖先保留各自字形，**单一固定模型无法兼得**，故提供该开关。代价：`arabic` 下同一章在标题行显示 `一`、在子节号里显示 `1`（同号不同形）——这正是中文书的排法，属预期。

**这些字段如何组合**——某级标题最终写入的前缀为：

```
继承前级 = 开：  前缀 + 父级序号（以序号间隔符拼接）+ 序号间隔符 + 本级序号 + 后缀 + 标题间隔符
继承前级 = 关：  前缀 + 本级序号 + 后缀 + 标题间隔符
```

> **继承前级用途：** 当某级（如 H5）希望独立编号（`a)`、`b)`）而非延续父级（`1.1.1.a`）时，关闭该级的「继承前级」即可。关闭仅影响**本级自身**的前缀拼接，不影响其子级是否继承本级。典型示例：
>
> ```
> H2  →  1            （继承前级=开）
> H3  →  1.1          （继承前级=开）
> H4  →  1.1.1        （继承前级=开）
> H5  →  a)           （继承前级=关，仅本级字母序号）
> H5  →  b)           （继承前级=关）
> ```

通过组合这些字段即可覆盖常见排版需求，例如：

```
前缀  序号样式     序号间隔符  后缀  标题间隔符   继承前级  →   写入效果
────  ──────────  ─────────  ────  ──────────  ───────       ──────────
（空） cjk         （不适用）  （空） 、           开        →   一、标题
（空） cjk         （不适用）  （空） （空格）      开        →   一 标题
（空） cjk         （不适用）  （空） . （点+空格）  开        →   一. 标题
（空） arabic      .          （空） （空格）      开        →   1.1 标题
（空） arabic      .          （空） . （点+空格）  开        →   1.1. 标题
（空） lower-alpha （不适用）  （空） )            关        →   a) 标题
第    arabic      .          章    （空格）      开        →   第1.1章 标题
```

> 注 1：序号间隔符仅在存在父级拼接时显现（H2 无父级，故 `一、` 不含间隔符；H3 起才出现 `1.1`、`1-1` 中的间隔符）。
>
> 注 2：**后缀**位于完整序号之后、标题间隔符之前，作用于本级**完整序号**（含继承的父级序号），故「第」+「章」得 `第1.1章` 而非 `第1章.1章`；与前缀配合实现「第N章」式编号。

**模板文件存储位置：**

每个模板保存为插件文件夹下 `templates/` 子目录中的独立 `.json` 文件。**首次启用插件时**，插件自动创建该目录并写入内置默认模板 `default.json`（在面板中显示为"默认"）。

```
.obsidian/plugins/auto-headings/
├── main.js
├── manifest.json
├── data.json               ← 插件设置（全局开关、路径规则、防抖延迟等），不含模板内容
└── templates/              ← 首次启用插件时自动生成
    ├── default.json        ← 内置默认模板（始终存在，显示为"默认"）
    ├── 学术风格.json
    └── 技术文档.json
```

文件名由模板名称经过路径安全化处理后生成（非法字符替换为 `-`，空格保留或替换为 `_`，保证跨平台兼容）。`data.json` 中的路径规则通过模板名称（即文件名去掉 `.json` 后缀后还原）引用对应模板。


**模板文件格式（`.json`）：**

```jsonc
{
  "name": "学术风格",
  "levels": {
    "h1": { "prefix": "", "numeral": "arabic", "suffix": "", "numberSeparator": ".", "titleSeparator": " ",  "inherit": true },
    "h2": { "prefix": "", "numeral": "cjk",    "suffix": "", "numberSeparator": ".", "titleSeparator": "、", "inherit": true },
    "h3": { "prefix": "", "numeral": "arabic", "suffix": "", "numberSeparator": ".", "titleSeparator": " ",  "inherit": true },
    "h4": { "prefix": "", "numeral": "lower-alpha", "suffix": "", "numberSeparator": ".", "titleSeparator": ") ", "inherit": true },
    "h5": { "prefix": "", "numeral": "lower-alpha", "suffix": "", "numberSeparator": ".", "titleSeparator": ") ", "inherit": false },
    "h6": { "prefix": "", "numeral": "arabic", "suffix": "", "numberSeparator": ".", "titleSeparator": ". ", "inherit": true }
  },
  "whitelist": [
    { "text": "目录",     "match": "exact" },
    { "text": "附录",     "match": "subtree" },
    { "text": "参考文献", "match": "exact" }
  ],
  "skipFill": { "mode": "fill", "placeholder": "0" },
  "topLevel": 2,
  "bottomLevel": 6
}
```

字段说明（字段顺序与磁盘上 `serializeTemplate` 的输出一致）：

- `name`：模板显示名（`default.json` 固定显示为"默认"）。
- `levels.h1`…`levels.h6`：H1–H6 各级，每级六个结构化字段——`prefix`、`numeral`、`suffix`、`numberSeparator`、`titleSeparator`、`inherit`（含义见上表；`suffix` 缺省为空、`inherit` 缺省视为 `true`）。`h1` 始终存在，是否对其编号由 `topLevel` 决定。
- `numeral` 合法枚举：`arabic`、`cjk`、`circled`、`lower-alpha`、`upper-alpha`、`lower-roman`、`upper-roman`（0.6.3 起含罗马数字）。
- `whitelist`：本模板生效时使用的白名单条目数组；每个条目含 `text`（词语）与 `match`（匹配方式，枚举 `exact`/`partial`/`subtree`，缺省视为 `exact`）（见 [3.7](#37-白名单系统)）。
- `skipFill`：**模板级**跳级策略——`{ "mode": "fill", "placeholder": "0" }`（补位，占位仅限数字）、`{ "mode": "drop" }`（不补位）或 `{ "mode": "none" }`（跳级标题完全不编号，0.7.15）；缺失/非法回退默认补 `0`（见上表与 [3.6 模板级字段](#36-模板系统)）。
- `topLevel`：**模板级**起始编号层级（1–6，缺失回退 H2）；比它浅的标题不编号、不被改写（见 [3.4](#34-起始编号层级与多个-h1)）。
- `bottomLevel`：**模板级**结束编号层级（1–6，缺失回退 H6 = 无下界）；比它更深的标题不编号、不被改写。与 `topLevel` 配合编号一个区间（0.6.3→0.6.5，见 [3.4](#34-起始编号层级与多个-h1)）。

**默认模板（`default.json`）的内容**——纯阿拉伯多级点分（即 `1` / `1.1` / `1.1.1` …）：

```jsonc
{
  "name": "默认",
  "levels": {
    "h1": { "prefix": "", "numeral": "arabic", "suffix": "", "numberSeparator": ".", "titleSeparator": " ", "inherit": true },
    "h2": { "prefix": "", "numeral": "arabic", "suffix": "", "numberSeparator": ".", "titleSeparator": " ", "inherit": true },
    "h3": { "prefix": "", "numeral": "arabic", "suffix": "", "numberSeparator": ".", "titleSeparator": " ", "inherit": true },
    "h4": { "prefix": "", "numeral": "arabic", "suffix": "", "numberSeparator": ".", "titleSeparator": " ", "inherit": true },
    "h5": { "prefix": "", "numeral": "arabic", "suffix": "", "numberSeparator": ".", "titleSeparator": " ", "inherit": true },
    "h6": { "prefix": "", "numeral": "arabic", "suffix": "", "numberSeparator": ".", "titleSeparator": " ", "inherit": true }
  },
  "whitelist": [
    // M3 占位三条；Milestone 4 扩充为中英常用结构性标题词表（见 3.7）
    { "text": "目录",     "match": "exact" },
    { "text": "附录",     "match": "exact" },
    { "text": "参考文献", "match": "exact" }
  ],
  "skipFill": { "mode": "fill", "placeholder": "0" },

```

选用结构化 JSON（而非自由格式字符串）的理由：与 GUI 列一一对应、所见即所得、无歧义解析；TypeScript 原生支持、无需额外依赖、Obsidian 的 `app.vault` 文件 API 可直接读写。

**序号样式枚举：**

| `numeral` 值 | 呈现 | 序列 |
|------|------|------|
| `arabic` | 阿拉伯数字 | 1, 2, 3, … |
| `cjk` | 中文数字 | 一, 二, 三, … |
| `circled` | 带圈数字 | ①, ②, ③, … |
| `lower-alpha` | 小写字母 | a, b, … z, aa, ab, … |
| `upper-alpha` | 大写字母 | A, B, … Z, AA, … |

> 跨级拼接时，**每一级父级各自套用其所在级别的 `numeral` 样式**（内部计数器仍是纯阿拉伯整数，样式仅在写入时套用）。例如 H2 用 `cjk`、H3 用 `arabic`，则 H2 显示 `一`、`二`，H3 的前缀里 H2 也呈现为中文，得 `一.1`、`二.1`。若再把 H4 设为带圈，则 H4 得 `一.1.①`——父级样式逐级向下可见。

**默认模板**（`default.json`）始终存在且不可删除，但可编辑。其他具名模板可在设置 GUI 中创建与删除——模板名称会出现在所有需要引用模板的下拉框中。重命名模板时，插件自动将旧文件删除并以新名称写入，同时更新 `data.json` 中所有引用该模板名的路径规则。

**删除被路径规则引用的模板——知情确认 + 安全降级。** 删除一个**未被任何路径规则引用**的模板时直接删除、不打扰。若该模板**正被路径规则引用**，则弹出确认对话框：**列出**受影响的全部规则，并让用户选择删除后这些规则的去向——降级到「默认」（缺省项）、改用其他指定模板、或连同这些规则一并删除：

```
┌─ 删除模板「学术风格」 ───────────────────────────┐
│  以下 2 条路径规则正在使用此模板：               │
│    · 读书笔记/                                   │
│    · 读书笔记/深度工作.md                         │
│                                                  │
│  删除后这些规则改用：[ 默认 ▼ ]                  │
│      （下拉可选其他模板，或选「删除这些规则」）   │
│                                                  │
│               [取消]    [确认删除]               │
└──────────────────────────────────────────────────┘
```

> 此设计兼顾**知情**（删前看见后果，绝不留悬空引用）与**省事兜底**（默认降级到「默认」，嫌烦直接确认即可），并保留高自由度（可改投他模板或连规则一起删）；既不像「静默降级」那样让用户事后才发现规则被改，也不像「禁止删除」那样强迫先手动解绑。

> **删除模板不回滚已格式化文件。** 删模板（及删路径规则）是**配置层**改动，插件**绝不**借此扫描全库去改文件——已写入文件的编号前缀**原样冻结**，直到该文件下次被触发重排（届时用新命中的模板覆盖）或被「清除编号」处理（见 [3.10](#310-清除编号)）。这与「插件只在触发时改当前活动文件、从不批量扫库」的核心写入模型一致（见 [3.3](#33-写入文件模型)），避免删个配置就造成大面积、难撤销的文件改动。

### 3.7 白名单系统

白名单是**模板级**配置：每个模板自带一份白名单，随该模板对文件生效（即由 [3.8](#38-按路径配置) 的路径解析决定当前文件使用哪个模板，就用哪个模板的白名单）。每个白名单条目由**词语**与**匹配方式**两部分组成；比较前对标题文本与条目词语应用同一套**归一化**（见下「匹配归一化」），归一化**仅用于命中判定，绝不改变写入文件的内容**。

匹配到白名单的标题：**不写入编号前缀**，且**不占用计数器槽位**（计数器既不累加也不归零，因此不产生跳号）。

**三种匹配方式（面向用户的唯一选择，按豁免范围递增）：**

| 匹配方式 | `match` 值 | 命中判定（归一化后） | 豁免范围 | 示例（条目「附录」） |
|----------|-----------|----------------------|----------|----------------------|
| 全部匹配 | `exact` | 与条目**完全相等** | 仅该标题自身 | 仅 `## 附录` |
| 部分匹配 | `partial` | **包含**条目子串 | 仅该标题自身 | `## 附录 A`、`## 技术附录` |
| 子树匹配 | `subtree` | 与条目**完全相等** | 该标题**及其整棵子树** | `## 附录` 及其下属 H3/H4… |

> **为何这三种够用：** 在「命中谁」上 `全部 ⊆ 部分`（部分是全部的超集）；`子树` 则在「全部」的精确命中之上，把豁免**向下扩展到子标题**。三者恰好覆盖三类真实诉求：只豁免这一个标题 / 凡含此词的标题都豁免 / 这一节整块都不编号。
>
> **注意「部分」与「子树」是正交的两件事**——前者放宽「命中谁」、后者扩大「豁免到哪」。首版刻意**不**提供「部分 + 子树」组合（即「凡含此词的标题连子树一起豁免」），以保持单选下拉的直观；确有需要再于后续版本以高级选项补充。

> **子树匹配范围：** 以**精确命中**的标题为根，覆盖其后所有**层级更深**的标题，直到遇到与根**同级或更高级**的标题为止。根标题自身与范围内全部子标题均不写前缀、不占计数器槽位。

> **子树块之后的计数器（决策 D1，0.7.10 已实现）：** 子树豁免块视为**独立结构**（如附录），
> 其后的标题编号**重新开始计数**——而非沿用块前的计数（旧行为，会得到令人困惑的 `3.2`）。
> 依据：对学术（APA/Chicago/IEEE）、技术（RFC 等）、中文文献（GB/T 7714）等规范的调研，**约 85% 的
> 文档范式中附录与正文编号断开、其后重新起算**；唯一例外的法律文书连续编号但根本不用子树白名单。
> 故**不设开关**、统一重置，GUI 白名单说明已注明。实现：`computeWhitelistExemptionDetail` 标出
> 子树块成员，引擎在块后 `counter.reset()`；夹在块与下一编号标题之间的 exact/partial 豁免不打断
> 重置。测试见 testplan D9/D10。
> **注意**：这只改「子树」豁免；「全部/部分」豁免仍是不占槽位、不重置（单标题豁免无独立结构语义）。

**匹配归一化（健壮性核心）。** 比较前对标题文本与条目词语应用同一套归一化，**仅用于判定，不改写文件**：

1. 剥离本插件写入的编号前缀（见 [3.5](#35-内部计数模型)）。
2. 去除行内 Markdown / HTML 标记：`**`、`*`、`_`、`` ` ``、高亮 `==文字==`、删除线 `~~文字~~`
   直接删除；HTML 标签（含属性，如 `<u>`、`<font color="#c3d69b">`、`<span style="...">`）整体
   剥离、保留标签内文字；并将链接 `[文字](url)` 还原为「文字」（HTML 标签 / `==`/`~~` 为 1.0.6
   新增，此前仅剥 `**`/`*`/`_`/`` ` ``/链接，见 D11）。
3. Unicode **NFKC** 归一（折叠全角/半角等价字符）。
4. 双侧 `trim()` 并将内部连续空白折叠为单个空格。
5. 拉丁字母**大小写不敏感**（统一转小写，使「Appendix」≡「appendix」）。

如此 `## **目录**`、含全角空格的 `## 目录`、`## APPENDIX` 都能稳定命中，杜绝「肉眼相同却不命中」。

**多条目命中同一标题——取豁免范围最大者（良定义的并集）：** 一个标题只要被任一条目命中即豁免；命中它的条目中只要有一条是**子树**（以它为根精确命中），它便连同整棵子树一并豁免。就标题自身而言「全部」与「部分」行为相同（差别只在「命中谁」），故实际效果等价于按范围求并集 `子树 > 全部 = 部分`——是并集，而非模糊的强弱比较。

**对「含子标题的标题」用全部/部分的告警。** 全部/部分只豁免标题**自身**；若它下面还有子标题，这些子标题会继续沿用**上一个已编号祖先**的计数（例如 `## 目录`〔全部〕下的 `### 小节` 会编成 `1.1`、错挂到前一节「第一章」名下）——这几乎不是用户想要的。故编辑面板对「自身被全部/部分豁免、却含子标题」的条目显示 ⚠ 提示，引导改用**子树**；引擎本身不做隐式改写（自豁免父标题没有自洽的子标题编号语义，这正是「子树」存在的理由）。

**实时命中预览。** 编辑面板针对**当前活动文件**实时列出「本白名单将豁免的标题」（命中数 + 标题清单），让用户在写入文件**之前**就看清每条规则、尤其是部分/子树的影响半径。

**默认模板预填充条目**（随 **Milestone 4** 内置；用户可在编辑面板中增删，默认均为「全部匹配」，可按需改为部分/子树）。覆盖常见的结构性/非内容标题，中英各一组——因匹配大小写不敏感，`References` ≡ `references`：

| 概念 | 中文 | English |
|------|------|---------|
| 目录 | 目录 | Contents |
| 附录 | 附录 | Appendix |
| 附图 | 附图 | Figures |
| 附表 | 附表 | Tables |
| 参考文献 | 参考文献 | References |
| 致谢 | 致谢 | Acknowledgements |
| 摘要 | 摘要 | Abstract |
| 索引 | 索引 | Index |

> 默认用「全部匹配」是最不具破坏性的选择（只豁免恰好同名的那一行）。`附录`/`Appendix` 这类常带子节点的，可在面板里改成「子树」整块豁免；要顺带覆盖 `附录 A`、`Appendix B` 等，可改成「部分」。

**编辑方式（0.7.16 起为行布局，testplan L16–L19）：** 在模板的展开编辑面板内提供白名单输入框——键入词语后按 Enter 添加为一条（默认「全部」匹配）；条目以**行列表**呈现，每行 = 词语（点击**行内编辑**，Enter/失焦提交，空 / 未变更 / 重复还原）+ 匹配方式**分段控件**（`=` 全部 / `≈` 部分 / `▸` 子树，三项常驻、单击切换、选中高亮，tooltip 双语）+ 命中数角标（tooltip 列出命中标题，超 8 条截断）+ ⚠ 含子标题告警 + `✕` 删除；完全相同的 (词语, 匹配方式) 自动去重。行上方有搜索 + 排序工具栏（M8 批次 1，纯视图不改存储；0.7.17 起默认按 A–Z 排序）。曾评估 `.gitignore` 式前缀语法的「文本模式」批量编辑，**不做**：与模板 JSON 可直接手改重复，且词语以 `=`/`~`/`>` 开头时有转义歧义。

```
白名单：[ 输入后按 Enter…   ]   目录[全部▼]✕   附录[子树▼]✕   参考文献[全部▼]✕
当前文件将豁免 3 个标题：目录 · 附录 · 附录 › 数据表
```

> **关于覆盖：** 白名单不再有"全局/文件夹/文件"三层作用域与并集开关。覆盖通过模板覆盖实现——路径专属模板会整体取代全局默认模板，连同其白名单一并替换。

> **配置不入 frontmatter。** 模板、白名单、路径规则均专属于插件设置 GUI 管理。插件对 frontmatter 的唯一交互是读取单文件开关键 `obsidian-auto-headings`（见 [3.2](#32-单文件开关frontmatter)）。

### 3.8 按路径配置

路径规则在插件**设置 GUI** 中管理——**一张可视化表格**，每行将一个路径模式映射到所选模板（通过下拉框选择）。某路径命中专属模板时，该模板**整体覆盖**（包含其各级格式与白名单）。

**根规则 `/` 即「全局默认」。** 不再单设「全局默认模板选择器」；取而代之，表格里一条**路径模式为 `/` 的规则**充当全库兜底——它匹配仓库下所有文件，是具体度最低的规则。首次启用时插件预置一条 `/` → 「默认」规则，于是开箱即对全库生效；用户可把它改投任意自定义模板，也可**整行删除**——删掉后即「无全局兜底」，只有命中更具体规则的文件才编号，自由度交还用户。（「默认」模板本身始终存在、不可删，但**引用它的 `/` 规则可删**——两者是不同的东西。）

**GUI 布局：**

```
┌─ 路径规则 ──────────────────────────────────────────────────────────┐
│  ⚠ 无根路径规则（/），「全局自动编号」开启时部分文件将不被编号  │
│     [ + 添加 / 根规则 ]   ← 仅当不存在 / 规则且全局自动编号=开时显示  │
│                                                                     │
│  [+ 添加规则]                                                       │
│                                                                     │
│  #  │ 路径模式                   │ 模板                      │      │
│  ───┼────────────────────────────┼──────────────────────────┼───── │
│  1  │ /                          │ [默认       ▼]           │ [✕]  │
│  2  │ Projects/                  │ [技术文档   ▼]           │ [✕]  │
│  3  │ 读书笔记/                  │ [学术风格   ▼]           │ [✕]  │
│  4  │ 读书笔记/深度工作.md       │ [简洁风格   ▼]           │ [✕]  │
│                                   ↑ 可拖拽排序                      │
└─────────────────────────────────────────────────────────────────────┘
```

> **路径输入建议弹窗（1.0.3 重做，testplan K13）：** 路径模式输入框接一个自绘的建议弹窗（`src/settings/tabs/PathSuggest.ts`）——随输入模糊匹配 vault 内**全部**文件夹 / 文件并排序展示（命中位置越靠前越优先，位置并列时文件夹优先于文件），支持键盘 ↑↓/Enter/Esc 与鼠标点击/悬停；选中文件夹自动带上尾斜杠。**参考实现：** 弹窗交互与「输入框 + 文件夹建议」的匹配思路参考自 numeroflip/obsidian-auto-template-trigger 的 `FolderSuggest`/`TextInputSuggest`（挂在 `activeDocument.body` 上、`position: fixed` 定位，不受 `.ah-path-table` 自身纵向滚动裁剪影响，比原生 `<datalist>` 更可控）。**用户报告 bug 的根治：** 本插件把「文件夹规则」与「文件规则」的区分系于路径末尾是否带 `/`，手动输入时极易漏打（填 `Foo` 会被当成对一个不存在的同名文件的精确匹配规则，行为上像是「改了模板没生效」）；除建议弹窗选中文件夹时天然带斜杠外，`autocompleteFolderSlash`（`pathrules.ts`）在提交时兜底：只要输入去除前导斜杠后与某个**真实存在**的文件夹路径完全相等却缺尾斜杠，自动补全。

> **列表可滚动 + 行内清空（M6）：** 路径规则过多时表格区可纵向滚动（表头吸顶），不再撑满整页看不全；每行路径输入框右侧有一个 **✕「清空此路径」**小按钮（只清空该输入框文本，**不**删除整条规则），与行尾**删除整条规则**的 ✕ 区分开。后者由按钮改为纯文本 ✕ 图标，不再被椭圆按钮背景包住。

> **兜底缺失提示。** 当表格中**不存在 `/` 根规则**、且「全局自动编号」为开时，面板顶部显示一条提示条「无根路径规则，部分文件将不被自动编号」，并附 **[+ 添加 / 根规则]** 快捷按钮——让「一堆文件突然不编号」这个状态**可见、可一键修复**，而非让用户困惑。这只是提示，不强制：用户确实想「只在特定路径编号」时无视即可。

**解析逻辑：**

1. 收集所有路径模式与当前文件匹配的规则（`/` 根规则匹配所有文件）。
2. **最具体的匹配**优先（精确文件路径 > 最长文件夹前缀 > `/` 根规则）。
3. **具体度并列**时（如两条不同文件夹名恰好等长），**行号更大**（列表中靠后）的规则胜出——这只是无法排出优劣时的确定性兜底，**不是**鼓励用法。
4. **若无任何规则匹配**（含 `/` 根规则已被删除的情形）：该文件**没有可用模板**。此时——**自动触发**静默跳过（不编号、不打扰、已有编号冻结不动）；**手动命令**弹 Notice「当前文件未匹配任何路径规则，无法编号」。frontmatter `true` 也无法凭空变出模板，故同样静默跳过（`true` 覆盖的是「触发资格」，不是「模板来源」）。

> **未配置行不等于根规则。** [+ 添加规则] 新增的一行路径模式初始为空串，代表「尚未填写」，**不匹配任何文件**——与用户显式填 `/` 表达「我要根规则」严格区分。曾经二者被等同处理，导致新增一行后只要先选了模板（尚未来得及输路径），就会被当根规则立即套用到全库并触发编号（testplan K11，0.7.22 修复）。

> **路径模式禁止重复（0.7.23）。** 两条规则的路径模式**归一化后完全相同**（含两条都填 `/`）却指向不同模板时，编号结果会取决于哪条「更靠后」这种用户在面板上看不出的内部顺序，体验上等同于随机、且用户报告确实造成过困惑。故 GUI 在提交路径编辑（新增一行填路径，或修改既有行的路径）时**主动阻断**：一旦归一化后与列表中其它某行重复，输入框回退为改前的值并弹 Notice「该路径已被第 N 条规则使用……」，不写入、不触发编号，逼用户先解决冲突（改路径或删掉其中一条）。这条约束**不分根/文件夹/文件**，任何路径模式的重复都拦；`resolvePathRule` 里"具体度并列时列表靠后者胜出"的兜底逻辑仍保留（应付历史遗留或手改 `data.json` 产生的重复数据），但不再是常规操作下会走到的路径。


### 3.9 触发与防抖

```
用户在编辑器中输入 / 粘贴 / 删除
        ↓
Editor onChange 事件触发
        ↓
防抖计时器重置（默认 300 ms，可配置范围 50–2000 ms）
        ↓  [计时器到期]
自动触发判定（见 [3.1](#31-开关命令与生效判定)）：「全局自动编号」=开 或 fm=true？且 fm≠false？路径规则命中模板？文件含标题？
        ↓  [全部满足]
完整文件重新编号周期启动（仅增删编号前缀，不改写任何 # 层级）
        ↓
以单一事务更新编辑器内容
```

- 防抖计时器以文件为单位；编辑另一个笔记不会取消当前笔记的待处理更新。
- 若文件在计时器触发前被关闭，待处理的更新将被取消（不向已关闭的文件写入）。
- 命令面板中提供**「立即重新编号」**命令，可绕过防抖立即触发；它走**手动命令路径**（不看「全局自动编号」开关、不被 fm `false` 拦截，仅受「能否命中模板」约束，见 [3.1](#31-开关命令与生效判定)），其余重排逻辑与实时触发一致。

**打开文件即重排（0.7.24，testplan J9）。** 除了编辑触发，`file-open` 事件也会立即（不经防抖）走一次**自动路径**判定——用户改了路径规则所投模板、或直接改了模板本身的样式后，该路径下**尚未编辑过**的文件不必等用户敲一下键盘，只要**打开**就自动刷新为当前生效格式。门控与编辑触发完全一致（「全局自动编号」=开 或 fm=true，且 fm≠false，路径规则命中模板）；已是最新格式的文件打开时，重排前后内容相同，不发起事务，静默 no-op，不会给每次打开都添一条撤销记录。

**外来编号自动路径守卫（1.0.7，testplan J10）。** 方案A下自动路径只认自己写的（WJ）前缀，对无 WJ 的标题一律当正文、直接叠加编号——这对**从其他编号插件 / 手写编号迁移**过来的文件很不友好：装上插件、全局自动编号一开，打开 `## 1 红米` 这类文件会被叠成 `## 1 1 红米`，观感上与 bug 无异，而这恰恰是新用户最容易撞上自动编号的时机（见 3.10「清理非本插件的标题编号」的写作动机）。为此在三处自动写入前（`scheduleRenumber` 的防抖到期回调、`renumberOnOpen`、`renumberActiveFile`）加一层只读探测 `hasUnclaimedForeignNumbering`（`cleanup.ts`）：

- **命中条件（两者都满足）**：① 全文完全不含 Word Joiner（= 插件从未给这份内容写过编号，是「第一次接触」）；② 用 `stripForeignNumbering` 探测到至少一个标题「像外来编号」。
- **命中后的行为**：跳过本次自动写入（不产生双重编号），改为提示「请先执行『清理非本插件的标题编号』」；同一文件本次会话仅提示一次（内存 `Set`，随文件改名迁移键、随删除移除、插件卸载清空），此后静默持续跳过，直至用户清理或重载插件。
- **只影响自动路径**：一旦文件已含任意 WJ（哪怕只有一个标题被本插件编号过），说明插件已"接管"该文件，此后即便又粘贴进一段带外来编号的新内容，也不再触发守卫——按方案A既有语义处理（该段落被当正文、叠加编号），这是已知边界、不在本次修复范围内，避免为"部分接管"引入更复杂的按标题级判断。
- **手动命令不受影响**：「立即重新编号」「清除当前文件编号」「清理非本插件的标题编号」都不查此函数，绕过一切开关照常执行——与既有「Renumber now 绕过一切开关」原则一致，用户显式触发的操作永远按既定语义执行。
- **已知风险**：探测器与 `stripForeignNumbering` 共享同一误伤面（如 `## API 设计` 会被判定为「像外来编号」），但落在这里代价是「跳过写入 + 提示」而非「内容被吃」，比清理命令本身更安全，故接受。

### 3.10 清除编号

编号是写进文件的**真实文本**，因此单靠删模板、删路径规则或取消勾选（frontmatter `false`）都**不会**抹掉已有编号（它们只停止后续自动编号，已有前缀**冻结现状**）。当用户确实想让某文件回到「无编号」的裸标题时，需要一个**主动剥离**动作。

**三个入口：**

| 入口 | 范围 | 行为 | 防误触 |
|------|------|------|--------|
| 命令「清除当前文件编号」 | 仅当前活动文件 | 剥**全部**编号（含插件自己写的 WJ 编号），还原裸标题 | 直接执行（单文件、可撤销，风险低） |
| 命令「清理非本插件的标题编号」（**0.6.6 新增**） | 仅当前活动文件 | **只剥「不含 WJ」的手写 / 外来编号**，**保留**插件自己写的（带 WJ）编号 | 直接执行（单文件、可撤销） |
| 面板按钮 **[清除全库编号]** | 全库所有 `.md` 文件 | 剥全部编号 | **二次确认对话框**；刻意**不注册为命令**，避免快捷键 / 命令面板误触发大面积改动；所在的「危险区域」**默认折叠**（M6），点击标题才展开，进一步降低误触 |

> **「清理非本插件的标题编号」为何而生（0.6.6，配合方案A）：** 方案A（见 [2.4](#24-保留正文起头的数字方案a)）下，插件**只管理自己写的（WJ 标记）编号**，对用户**手写 / 导入**的伪编号（`## 1. X`、`## 第三章 X`、`## 1.2 X`）一律当正文、不再吸收。于是用户导入带手写编号的文档时，需要一个主动命令把这些**外来编号**清掉，好让插件接管。该命令逐标题判断：**含 WJ → 本插件的，不动**；**不含 WJ → 用更广的手写编号剥离器**（`stripForeignNumbering`：覆盖 arabic/cjk/带圈/字母/罗马 + `第…章` + 成对括号 `(1)`/`（一）`/`[1]`/`【1】` + `1.`/`1)`/`一、` 等惯例）剥一层。**已知风险**（与「清除编号」同源、属预期）：以序号样字开头紧跟分隔符的真实标题可能被误剥（`API 设计`→`设计`、`2024 总结`→`总结`）——本命令是用户**主动**一次性操作，已接受。

**行为：** 逐个标题剥离本插件写入的编号前缀、**不再写入新前缀**，把标题还原为裸标题；其余文本不动。剥离以**单一事务**写回（全库版逐文件写回）。

**剥离器——「全样式并集」正则，独立于模板与 WJ。** 这是关键设计点：清除时文件可能**已无任何模板命中**（`/` 根规则、专属规则都被删了），没有「某个模板」可作剥离依据。故「清除编号」使用一个**不依赖具体模板、也不依赖 WJ** 的剥离器 `stripPrefixBroad`——它认得所有序号样式（arabic ∪ cjk ∪ circled ∪ 字母 ∪ 罗马）拼出的前缀 token，尽最大努力剥掉任何本插件可能写过的前缀（含 0.6.4 前无 WJ 的历史前缀、用户手写前缀）。**注意（方案A，0.6.6）：常规重排的 `stripPrefix` 已改为纯 WJ 边界（只剥带 WJ 的），而「清除编号」刻意仍用这个更激进的全样式正则**——因为「清除」的语义就是「把一切像编号的都抹掉」，与常规重排「只动自己写的」相反。「清理非本插件编号」命令则用介于两者之间的 `stripForeignNumbering`（更广的手写惯例正则，但只作用于**不含 WJ** 的标题）。

> **与 frontmatter `false` 的分工：** `false`（取消勾选）= 「以后别自动碰我」（冻结、不清除）；「清除编号」=「现在把现有编号抹掉」（一次性动作）。二者常组合：**先「清除当前文件编号」抹掉，再取消勾选** 防止下次又被自动编号。
>
> **已知风险（可接受）：** 若用户**手写**了形如 `1.1 标题` 的编号，全样式剥离器可能将其误当作插件前缀剥掉。这与现有 `stripPrefix` 的固有风险同源——[2.3](#23-边界情况与设计决策) 已认定「对编号前缀的手动编辑是预期行为」，故接受此权衡。

> **待修（上架审计发现，testplan H8）：** `clearAllVaultNumbering`（面板 [清除全库编号]）逐文件走
> `vault.read` + `vault.modify`，而非 M18 已验证更安全的 `vault.process` / 编辑器内存改写路径——若目标文件
> 此刻正被打开且有未落盘的编辑器改动，写回可能覆盖它，与 M18 修复前的竞态同源。尚未实测复现（未落盘改动
> 与全库清除同时发生的窗口很窄），暂列 backlog，不阻塞本轮发布。

### 3.11 国际化（中英双语，M6）

设置面板、命令名与 Notice 提示**全部支持中英双语**（`src/i18n.ts`）。

- **语言来源：** 设置面板顶部「语言」下拉，三选一——**自动（跟随 Obsidian）/ 中文 / English**，默认「自动」。
  「自动」读取 Obsidian 的界面语言（官方 API `getLanguage()`，1.8.0+，以 `zh` 前缀判为中文，其余英文；
  调用失败回退英文。1.0.1 起弃用早期的 `localStorage["language"]` 探测，商店审核要求）。选定后面板立即

- **文案结构：** `Messages` 接口列出全部界面字符串；`zh` / `en` 两套实现**形状完全一致**（TS 保证不漏键，

### 3.12 Backlink 同步（M7，1.0 发布前置）

> 「改标题/编号后 `[[file#heading]]` 链接失效」是社区
> **呼声第一**的痛点，目前仅 Header Enhancer 解决。本插件**每次编号都会改写标题文本**（加 / 改 / 去前缀），
> 正是该痛点的高发场景。故把 Backlink 同步定为 **1.0 上架前必做**，补齐唯一硬短板。

**问题。** 编号把 `## 简介` 改成 `## 1 简介`（或改样式 `一、简介` → `1 简介`，或清除还原成 `## 简介`），
任何指向旧标题锚点的内部链接——`[[file#简介]]`、`[[#简介]]`、`[[file#简介|别名]]`——都会断链。Obsidian
**不会**因标题文本变化自动修链（其自动修链只走「重命名文件 / 用命令重命名标题」流程，不覆盖插件直接改写正文）。

**参考实现：** 唯一解决此痛点的 **Header Enhancer**（hobeedzc/obsidian-header-enhancer-plugin 的
`src/backlinks.ts`，见「关于」TAB 鸣谢）的做法：命令触发编号后，
按**行号**配对「旧标题文本 → 新标题文本」（编号逐行就地改写、不重排行，故配对无需模糊匹配），
用 `metadataCache.getBacklinksForFile(target)` 取 `{data: Map<sourcePath, ReferenceCache[]>}` 反查引用方，
比对 `ref.link` 命中后**备份内容 → `vault.read` + `vault.modify` 写回 → 失败回滚**，opt-in 开关默认关。
我们**沿用其架构，并在它偷懒的四处做得更稳**（见下「改进点」）。

**设计原则。**
- **只在编号触发的同一次改写里同步**：编号 / 清除 / 清理外来编号 = 改动标题文本的全部来源，故 Backlink
  同步**挂在这些写回之后**，以「本次哪些标题文本从 X 变成 Y」为输入，不做独立的全库扫描循环。
- **逐行配对、无需模糊匹配**（承 Header Enhancer 洞察）：编号引擎逐行产出，旧 `parseHeadings` 与新
  `parseHeadings` 按 `lineIndex` 配对即得「旧→新」，零成本。
- **开关 `updateBacklinks`，默认开，全局生效**（曝光度决策，0.7.11；单开关合一，1.0.9）：内测期曾默认关
  （保守，与 Header Enhancer 一致）；上架前重估——它是 1.0 头牌卖点、且有「重复标题保守不改 / 块引用与
  多级锚点跳过 / 静默降级」护栏，改为**默认开**。开启后**与是否命中编号模板 / 是否实际写入编号无关**：
  标题文本一旦被改写就同步引用链接（1.0.9 前需额外勾选「独立触发」opt-in 才有此效果，见下方 CR-18
  段落的演变说明；现已合并为同一个开关，不再需要用户理解两层语义）。配套告知：**首次实际改写引用文件时
  弹一次说明 Notice**（说清改了什么、改动不在被改文件的 undo 步内、在哪里关；`backlinksIntroShown`
  持久化只弹一次）；面板上作为普通设置项自然呈现（紧跟「全局自动编号」，不做高亮）。迁移口径：旧
  data.json 缺失该字段视为开，显式设过 false 保留；历史独立开关字段 `backlinkStandaloneTrigger`
  不再读取，加载时清理。
- **锚点归一 = 匹配 / 写入双口径**（0.7.3 修，见 testplan M11）：
  - **匹配用 `linkAnchor`**（改名表 `from` 键 + 引用链接 subpath）：**剥 WJ** + 去 `[ ] # | ^` + 折叠空白 + trim，
    故无论既有链接是否含 WJ 都能匹配。
  - **写入用 `displayAnchor`**（改名表 `to`，即写进链接的文本）：同上但**保留 WJ**。编号写入的标题含不可见 WJ
    （`## 1 ⁠标题`），Obsidian 标题锚点解析按字节比对、**不剥 WJ**——剥了 WJ 的链接解析不到该标题、显示断链
    （实测确认）；保留 WJ 的链接与真实标题字节对齐 → 必然解析（裸标题无 WJ 时两者等价）。

**流程（每次标题文本改写后，`updateBacklinks` 开时）：**
1. **算改名表** `computeHeadingRenames(oldContent, newContent)`：两侧 `parseHeadings` 按 `lineIndex` 配对，
   取 `linkAnchor` 后 `from !== to` 且非空者入表；**重复锚点**（同名标题多处）从表中剔除（歧义不改，见下）。
2. **同文件内链就地折叠**（`foldSelfBacklinks`，0.7.25 实修，见下「同文件内链」）：把 `[[#锚点]]` /
   `[[本文件#锚点]]` 的改写直接对 `newContent` 做字符串重写，随本次编号/清除**同一个** `editor.transaction`
   一并写回——不经过 `vault.process`，故不读盘、不与本文件编辑器的未落盘改动竞态。
3. **反查引用方** `metadataCache.getBacklinksForFile(target)`：取 `.data` Map 的 sourcePath 列表（= 链向
   本文件的**别的**文件；本文件自身这一条已在步骤②处理，`syncBacklinks` 显式跳过避免重复/竞态）。
4. **重写锚点** `rewriteBacklinksInContent`（纯函数）：在每个引用文件里正则扫 `[[…]]` / `![[…]]`，对
   「路径段 basename 命中本文件、subpath 归一后落在改名表」的链接替换 subpath，**保留别名 `|alias` 与嵌入 `!`**。
5. **写回与反馈**：别的文件用 **`vault.process(file, fn)`** 原子 read-modify-write；本文件自身的命中数
   （步骤②）与别的文件的命中数相加，给一条 Notice（`已更新 N 处链接`）。

**比 Header Enhancer 改进的四处：**
1. **原子写入** `vault.process`（它用 `vault.read`+`vault.modify`，读改写非原子、可能覆盖并发编辑）。
2. **保留别名 / 嵌入**：它的 `backlinks.ts` 不区分别名与块引用；我们解析 `|alias`、`!` 嵌入并保留。
3. **重复标题消歧**：它按子串匹配易错改；我们对重复锚点**保守不改**（剔出改名表），避免错改，登记 testplan 已知限制。
4. **自动路径 gate**：它纯命令驱动；我们有防抖自动路径，故**仅当改名表非空才进入同步**（日常打字标题文本不变 → 零开销）。

**规避它暴露的两个风险：**
- `getBacklinksForFile` 是**半公开 / 未文档化 API**（返回 `{data}` 包装）：包一层适配 + 缺失时静默降级（不报错、不阻塞编号）。
- **不用子串匹配判定命中**：改走 `linkAnchor` 规范化的整体相等 + 路径 basename 命中，避免「含该词的别的标题」被误改。

**边界与开放问题（登记 testplan，部分留 backlog）：**
- ~~**纯文本改名不同步**~~（**0.7.8 已修**，testplan M14/M15）：改名表基线从「编号前」升级为
  **上次同步点快照**（`headingSnapshots`：文件打开时播种、每次插件写回后刷新，与开关无关地维护）——
  用户在两次触发之间做的纯文本改名（编号不变、甚至编号器无写回）也能被 `computeSnapshotRenames`
  看见。基线与现内容之间正文行可能增删，故按**标题顺序**配对；**结构变化**（同轮增删标题 / 改层级）
  时按序配对不安全，保守回退「编号前 → 编号后」口径——编号侧改名仍同步、文本改名当轮放弃
  （testplan M16 取舍；下轮结构稳定后恢复可见）。沿用同一 `updateBacklinks` 开关与快路径直接改写
  （审阅模式留待 M9，见 [Roadmap](#5-roadmap)）。
- ~~**WJ 与链接解析**~~（**0.7.3 已修**）：实测确认剥 WJ 的链接解析不到含 WJ 的标题（显示断链），已改为
  写入侧 `displayAnchor` **保留 WJ**（匹配侧 `linkAnchor` 仍剥 WJ）。注意：Backlink 同步只对**标题变更**
  作出反应（0.7.8 起含纯文本改名）——对「手敲时就不匹配的既有断链」不会主动修（无标题变更可触发，
  属设计取舍，testplan M13；全库扫描修复留待 M9，见 [Roadmap](#5-roadmap)）。
- **重复标题锚点**：同名标题多处时锚点歧义，本版**不改**（保守）；Obsidian 用 `#标题-1` 消歧的精确改名留 backlog。
- **多级锚点 `#A#B`、块引用 `^id`**：本版只改单段标题锚点，多级 / 块引用**跳过不动**（保守）。
- ~~**同文件内链 `[[#锚点]]`与编辑器缓冲竞态**~~（**0.7.25 已修**，用户实测报告：清除编号后 Notice
  提示已清除、文件却未变，切到别的文件再切回才生效）：根因是旧实现把「引用方=本文件自身」交给
  `syncBacklinks` 的 `vault.process` 处理——`vault.process` 读的是 vault 缓存 / 磁盘内容，而本文件此刻
  editor.transaction 刚写入、尚未被 Obsidian 自动保存，二者异步竞态，`vault.process` 读到旧内容写回
  覆盖掉刚做的编号/清除。改为 `foldSelfBacklinks`：本文件自身这一支直接对内存里的 `newContent` 做
  字符串重写、随原编号/清除**同一个事务**写回，不经过任何异步读盘，无竞态可言；`syncBacklinks` 只再
  处理**别的**引用文件（并显式跳过本文件自身，避免重复处理）。
- **撤销一致性**：引用文件的改写不在本文件编号事务的 undo 步内（同 Header Enhancer）；文档说明。
- **关开关后的历史断链**：开关只管「从此刻起」，不回溯；全库「扫描修复历史断链」命令已升格进 [M12](#milestone-12--独立价值包--infra-转向)。
- **独立于编号模板的触发（[M12](#milestone-12--独立价值包--infra-转向) 首项，1.0.8 落地为独立开关、
  1.0.9 并入单开关，2026-07-10 定位倒转，动机见 [附录 A](#附录-a--方向审查与否决理由grill) §1）**：
  Backlink 同步曾**完全耦合在编号触发链路里**——唯一入口 `applyRenumber`（`main.ts`）只被
  `scheduleRenumber`（需 `shouldAutoTrigger` 且 `getTemplateForFile` 命中模板）与 `runImmediateRenumber`
  （同样需要命中模板）调用，模板解析不到就完全不跑，Backlink 同步跟着不跑。但核心纯函数
  （`computeHeadingRenames` / `computeSnapshotRenames` / `rewriteBacklinksInContent`）从不依赖模板或
  编号样式，只认"某行标题旧文本→新文本"——耦合只在**触发入口**，不在逻辑本身。1.0.8 先实现为一条
  不依赖 `getTemplateForFile` 的触发路径（复用既有 `headingSnapshots` 快照基线判断"标题文本变了"，
  命中即走 `foldSelfBacklinks`/`syncBacklinks`、跳过 `renumberContent`），配一个独立开关
  `backlinkStandaloneTrigger`（默认关，opt-in）。实测发现**两层开关对用户是无谓的认知负担**——多数
  用户理解不了"总开关"与"独立触发"的区别，也说不清为什么要分两步才能让"改标题不断链"生效。1.0.9
  **将两个开关合一**：`backlinkStandaloneTrigger` 字段删除，其语义并入 `updateBacklinks`——开启后
  该触发路径**始终生效**（不再 opt-in），受一条优先级更高的保护约束：frontmatter 显式 `false`
  （用户对该文件的明确「别碰」表态）；`vaultClearInProgress` 清库进行中同样压制；常规编号路径已处理
  本轮时该路径不重复跑（testplan M19–M26）。产品意义：
  把插件从"标题编号"赛道里再分出"标题改名不断链"这个更少竞争的独立价值点，编号本身降级为可选进阶
  功能而非使用 Backlink 同步的前提，且用户只需理解一个开关。

  会触发已打开文件的 `editor-change`，不压制的话刚清掉的编号会被立刻编回去；完毕（含异常）恢复。
  **0.7.17 起清库前还会先持久关闭「全局自动编号」**（testplan H7）：清完却留着开关开，一编辑又被编
  回去——反直觉；清库 = 用户明确表态「现在不要编号」，故先关再清，需要时手动再开（双语 desc 与确认
  对话框已注明；frontmatter `true` 的文件不受全局开关约束，故清库期间的临时压制仍保留）。
- **TAB 图标化 + 窄屏滑动**（0.7.17，testplan L21/L22）：TAB 用 Obsidian 内置 lucide 图标
  （`setIcon`，SVG currentColor 黑白随主题；刻意不用 emoji——系统 emoji 字体固定彩色不吃 CSS）。
  未激活 = 仅图标（名称经 aria-label/title 提供），激活 = 图标 + 文字 + 主题强调色背景；
  TAB 栏 `overflow-x: auto` 自身横向滑动，窄屏绝不撑宽设置页。图标映射：全局设置 `settings`、
  路径模板 `folder-cog`、敏感操作 `alert-triangle`、关于 `info`。
- **模板编辑面板布局**（0.7.17，testplan L23）：折叠钮（▸/▾）**前置**作展开标志，模板名紧随其后、
  点击二者均可切换；展开时整行变**大框**扩住标题行与面板，面板缩进 26px 与模板名左对齐（折叠钮悬在
  左侧沟槽）；「跳级缺失层级 / 占位字符」上移到层级网格前与其余模板级下拉归组；网格装入「级别格式」
  子框、白名单整块子框，两大块对称。序号样式下拉标签精简为纯示例字形（「1, 2, 3」，L24）。
- **设置面板输入的 IME 感知**（0.7.17，testplan J8/L25）：拼音等组合输入期间（`isComposing`）
  文本输入框不提交、编辑器防抖到点顺延一个周期，`compositionend` 后再提交 / 写回——否则组合中间态
  （半截拼音）被逐键保存并编入标题。document 级 compositionstart/end 维护 `imeComposing` 标志。
- **默认模板显示名随语言**：存储名恒「默认」、文件名恒 `default.json`，仅 GUI 显示层按语言映射
  （「默认」/「Default」），路径规则引用与磁盘不变。
- **配套精简**：预览每级只显 2 例（原 3）；各设置说明**一句话化**（长解释下沉本 spec）；模板网格与
  路径表格在窄屏（移动端）横向滚动。
- **预览优先于文字说明（设计原则）**：能用渲染示例说清楚的地方不写说明文字，示例已经暴露的信息不用
  文字复述——既有先例是 `previewLevel()` 的逐级示例与白名单编辑器的实时预览。仅当预览无法表达"为什么
  这样设计"（如 `ancestorNumeral` 这类概念性取舍）时才保留一句话说明。**已知缺口**：`skipFill`（跳级
  占位策略 fill/drop/none）目前只有文字描述、没有配对的渲染示例，是这条原则的第一个待补项，见
  Roadmap M9。
- TAB 与展开态是**面板存续期**的视图态（重开面板回到「全局设置」），不持久化。

### 3.14 侧栏大纲导航（M8a）

> **状态：规划中，未落地。** 目标：提供原生风格的侧栏大纲面板，作为「唯一集编号维护 + 结构导航于
> 一体」差异化的第一阶段——**只做只读导航**（展示 / 定位 / 搜索 / 跳转），不含结构变更。结构变更
> （拖放重排、行内编辑）拆到 [3.15](#315-拖放重排与结构编辑m8b) 独立评估落地，理由见该节引言：结构
> 变更是新的写入路径、风险显著更高，不应拖累导航价值本身的交付。呈现形态选定为**侧栏树形列表**
> （非笔记边缘 minimap 式标记条）——评估过 Modern Outline 的 minimap 范式作为替代形态，两者是不同
> 的产品方向，不同时做；比较过程见 `doc/log.md` 本周期记录。

**UI 参考（设计稿）：**

**面板外壳：**
- 注册自定义 Leaf Type（`auto-headings-outline`），用户可在侧栏添加/删除/切换，与 Files/Backlinks 平齐
- 标题区极简：⚙️ 设置图标（lucide）+ 🔍 搜索框 + ✕ 清空按钮；⚙️ 点击快捷打开设置面板

**层级滑块：**
- 默认显示 H1–H6 六个节点，均匀分布；按当前文件实际存在的标题等级动态禁用（示例：文件只有
  H2/H3/H4 → 显示六个节点但 H1/H5/H6 灰显不可点）
- 滑块显示当前视口最高标题层级（默认 H2；超出存在范围自动降级到最深有效级），位置持久化到
  Settings，下次打开文件自动恢复。**约束**：这类"记住状态"只写进插件唯一的 Settings/`data.json`，
  不为每个文件生成单独的 side-car 状态文件——Quiet Outline 曾报告关闭"持久化状态"开关后仍在 vault
  里写状态文件、与 iCloud 同步产生重复文件的 bug；未来若扩展到"记住每个文件的展开节点"，也要沿用
  同一份集中存储，不重蹈这个坑
- **拖动滑块的过渡动画（决策）**：标题树展开/折叠 + 展开符号旋转 + 子标题淡入淡出，**先落地固定
  时长**（如 150ms），暂不做"按点击频率自适应 50–200ms"的版本——这类效果没有自动化验证手段（见本
  节末「测试策略」），规格阶段不必卡在这类细节上；留作面板正式可用后的可选打磨项，非阻塞

**标题树与搜索：**
- **标题层级解析直接复用 `parser.ts` 现有实现**（跳级 H3→H5 的补位/不补位/不编号三种模板级策略、
  围栏代码块边界判定等），面板只负责渲染这棵树、不重新实现一套独立的层级识别逻辑。对比：Quiet
  Outline README 里明确写了限制"不支持跨级标题（h1→h3→h4 会被当成 h1→h2→h3）"，根源正是它的大纲
  面板与编号逻辑是两套独立实现——本插件应避免"面板与核心解析器各写一套、行为不一致"的覆辙
- 缩进列表展示完整编号（如 `1.1.2 标题文本`）与原始 Hx 等级（轻显示）
- 当前视口位置自动定位：编辑器焦点所在标题自动高亮（淡主题色），侧栏自动展开路径直到包含该标题
- 搜索框实时过滤，只显示标题文本或编号匹配的行且**保留树形结构**（不 flatten）：搜索结果不 flatten，
  未匹配的兄弟分支自动折叠。例如搜索「方法」时，# 定期 / ## 方法 / ### 方法1 / ### 其他 展示前三个
  并折叠「其他」；祖先路径（# 定期）保持展开便于定位。有焦点时上下箭头逐个高亮匹配行、回车跳转、Esc 清空

**高亮：** 当前位置高亮与搜索匹配高亮均用淡化的 Obsidian 主题色、彼此可区分，浅色/深色主题下均清晰
可读。**非目标**：不按标题级别（H1/H2/H3…）分配不同颜色——Quiet Outline 的"彩虹色"配色是被社区自发
写 CSS 片段替换成主题色的真实痛点（见 `doc/log.md` 本周期调研记录），标题级别只用缩进 + 轻显示的
`Hx` 文本区分，颜色只服务于"当前位置"与"搜索匹配"这两种交互状态。

**其余交互（均为只读操作）：** 展开/收起符号点击、双击标题跳转编辑器对应位置、右键菜单（Copy link to
heading / Copy as Markdown list / Expand all / Collapse all）、键盘导航（Up/Down 切换、Enter 跳转、
Space 展开/折叠）、面板可横向滚动（深层嵌套）。跳转（双击 / Enter）后**焦点应还给编辑器**，不留在
面板里——Quiet Outline 一个被反复提及的体验抱怨就是点完标题后焦点滞留侧栏，用户得手动点回文档才能
继续打字；仅当用户正在搜索框内操作时焦点才停留面板。移动端本节只要求**展示**随侧栏宽度自适应；触摸
长按拖拽属于 M8b（见 [3.15](#315-拖放重排与结构编辑m8b)）。

**性能：** 编号变化时随 `editor:changed` 自动刷新；大文件（1000+ 标题）树渲染需虚拟化（仅渲染可见
区域）；搜索输入防抖 100ms，且**需比照 `main.ts` 现有的 `imeComposing` 模式**（`compositionstart`/
`compositionend` + 防抖顺延，见 [3.9](#39-触发与防抖)）与设置面板 `WhitelistEditor.ts`/`EditPanel.ts`
里对 `isComposing` 的处理——中文拼音等输入法组合期间不触发过滤，避免半截拼音被当作搜索词、打断输入
法（这是 main.ts 里已经解决过的问题，不要在新输入框上重新踩一遍）。

**测试策略（设计决策）：** 现状 `vitest.config.ts` 固定 `environment: "node"`，`obsidian-mock.ts` 的
`containerEl` 只是空对象，不产生真实 DOM（见该文件顶部注释「DOM 结构由手验覆盖」）——这套面板此前
没有任何自动化验证空间。计划：M8a 落地时给**新增的这一个测试文件**用 vitest 的按文件覆盖机制
（`// @vitest-environment jsdom`）引入最小 `jsdom` 依赖，不必把既有全部测试改到 jsdom、不拖慢现有用
例，即可对「树结构是否正确渲染 / 搜索过滤是否命中 / 键盘导航状态机是否正确」做真实断言。边界：jsdom
没有真实布局与动画引擎，CSS 过渡时长、fade in/out 视觉效果仍验证不到，与 testplan 现有原则一致（面板
类交互「无文本语义，留手验」）——这部分维持现状，交给 `user_tests` 手验。

### 3.15 拖放重排与结构编辑（M8b）

> **状态：规划中，未落地。** 本节是 M8 里风险最高的一段：把「原地重排标签」升级为「剪切—拼接—重排—
> 同步 backlink」的结构性编辑，是当前写入模型（见 [3.3](#33-写入文件模型) 与 [§4](#4-架构设计)「从不
> 批量扫库、整文件重写」不变量）之外的**新写入路径**，边界情况数量级预计超过 M6/M7 已加固的「原地
> 改标签」场景（白名单子树边界、代码块边界、多级锚点交叉后会更复杂）。因此独立于 [3.14](#314-侧栏大纲导航m8a)
> 单排 M8b，可在导航面板先行发布之后再评估，不阻塞前者。

**拖放重排与自动重编号：** 用户可拖动任意标题行到另一个标题下面（变子标题）、同级（改序号）、或提
到上级（改层级）。示例：源 `2.1 核心算法` 拖到 `1.2 定义` 下 → 结果自动变为 `1.2.1 核心算法`（层级 +
编号 + backlink 一并更新）。拖放后一次 transaction 完成：① 检测拖动源的 Markdown 块范围（含下级子标题）→
② 在源位置剪切该块 → ③ 粘贴到目标位置 → ④ 重新计算全文编号（层级变化 / 编号递进 / 白名单应用）→
⑤ 同步所有受影响 backlink → ⑥ 一次提交，可用 Ctrl/Cmd+Z 整体撤销。拖放进行期间临时禁用编辑器（避免
与用户并发输入冲突），完成或取消后自动恢复。

**边界约定：** 允许拖到自身兄弟处、拖到其他层级下、拖提升到上级；**不允许**拖到自己下面（保护循环
依赖）、拖到代码块内（代码块边界自动检测）、拖到白名单豁免子树内部——**决策：直接禁止，不做"允许但
弹警告"的中间态**。理由：白名单子树本就是"块级豁免、结束后计数器整体重置"的结构性边界（见
[3.7](#37-白名单系统) 决策 D1），与"不允许拖入代码块"是同一类不变量，没必要留成实现时才拍板的开放
问题。

**拖放视觉反馈：** 拖起时该行高亮 + 半透明；悬停有效目标显示"will insert as child"/"will reorder"
提示，悬停无效目标显示禁止符号；拖放完成后该标题短暂闪烁提示编号已更新。

**交互面选址（0.7.28 补充设计）：** 在原有"侧栏树内拖拽"之外，新增一个更贴近 Obsidian 原生体验的
入口——直接在**主编辑器的 gutter 区域**（Obsidian 原生标题折叠三角所在的位置）追加两个小控件：一个
升降级按钮对（∧升级、V降级，改变该标题自身的 `#` 层级）、以及一个拖放把手（把整个标题段——该标题及
其下属子标题——拖到文档内其他位置）。**不改变 Obsidian 原有的 gutter 布局**是硬约束，折叠三角的位置
与行为不受影响，新控件只是追加，不能遮挡或抢占它的点击热区。

这两个控件对应两种不同复杂度的操作，建议分开排期：
- **升降级按钮**：只改变一个标题自身的 `#` 数量，不涉及"剪切—拼接到任意位置"，复杂度远低于拖放。
  **未决问题（详细设计阶段需拍板）**：点击后是否要级联调整该标题下属子标题的层级以维持相对结构
  （如把 `## 2.1` 升级为 `# 2` 后，其下 `### 2.1.1` 是否也要跟着升一级变成 `## 2.1`，否则会出现"子
  标题比父标题深两级"的断裂结构）——这个问题不解决，升降级按钮就有制造出无意义标题树的风险，不能
  默认"只改这一行"。
- **拖放把手**：与既有"侧栏树拖拽"设计是同一套底层能力，[下方测试策略](#315-拖放重排与结构编辑m8b)
  里已经规划的"纯函数层 `moveHeadingBlock` + DOM 手势层"两层拆分依然成立，差别只是手势识别层挂在主
  编辑器的 CM6 gutter widget 上、而不是侧栏树的 DOM 节点上——纯函数层可以完全复用，不必重写。

技术上追加 gutter 控件需要用 CodeMirror 6 的 gutter extension（Obsidian 侧对应 `registerEditorExtension`），
与 Obsidian 自带的折叠三角是两套并存的 gutter provider，需验证二者能否正常共存渲染（间距、点击热区是否
互相干扰）、深浅色主题下是否都可读——这部分留待详细设计与实现阶段验证，规格阶段先记录这个技术风险点。

**行内编辑（inline edit mode）：** 单击标题行（非展开符号区）进入编辑态，展开为文本框，显示原始
Markdown 行但**隐藏编号部分**；用户只编辑标题正文，编号由插件维护、不可手改；支持 Markdown 格式
（`**bold**`/`_italic_` 等）与编辑器同款快捷键（Ctrl/Cmd+B 加粗、+I 斜体、+K 链接、+\` 代码）；Enter
提交 / Esc 取消 / 失焦自动提交；提交后若标题文本改变则重新编号 + 同步 backlink。

**移动端触摸拖放：** 用触摸长按触发拖拽，侧栏宽度自适应。移动端在更简单的既有功能上就出现过需要专
项排查的情况（M7 testplan MOB-1），触摸拖放是这里面复杂度最高的交互，预期需要独立的移动端实机验证
周期，不与桌面端同批验收。**建议**：移动端触摸拖拽作为 M8b 桌面端交付之后可独立延后的范围，不阻塞
M8b 本身的验收——调研两个参照对象（Obsidian 核心 Outline 面板、Obsidian Outliner 插件）均未在"标题
拖拽"这个复杂度上做出稳定的移动端体验，没有前人经验可抄，硬凑进同一批验收只会拖累桌面端功能上线。

**测试策略（设计决策，关键架构建议）：** 建议把「结构变更执行」与「拖放手势识别」显式拆成两层，参照
现有 `parser`/`numbering` 与 `main.ts` 的关系：
- **纯函数层**（新增，如 `moveHeadingBlock(content, source, target): string`）：输入整篇 Markdown +
  被移动标题的范围 + 目标位置，输出重排后的整篇内容；不碰 DOM，不知道"拖拽"这回事。
- **DOM 手势层**（面板内的 dragstart/dragover/drop 处理）：只负责识别用户想把哪个标题拖到哪里，算出
  source/target 后调用上面的纯函数。

这样拆分后，纯函数层可以按 `tests/dev_tests/uvm/`（约束随机序列，见 [§4 随机序列压测](testplan.md)）
现有模式**新增一种随机操作**（"随机选两个标题做移动"），配套不变量（移动后标题总数不变、层级与编号
仍自洽、白名单子树未被破坏、backlink 改名表如预期同步）——这是 M8 里少数能被机器持续回归、而非仅靠
人工点击验证的部分，建议在实现前就定下这个拆分，而不是先写成一个揉在一起的 DOM 事件处理函数再回头拆。
真实拖拽手势的触发时机、touch 长按阈值、跨浏览器 DataTransfer 行为，与动画一样不在 jsdom/UVM 覆盖范
围内，仍需 `user_tests` 手验。另需显式补一条 `user_tests` 场景：**在带 frontmatter 的文件上做拖
放**——Obsidian 核心 Outline 面板历史上出现过"笔记带 frontmatter 时拖放失效"的回归（v1.4.5），而本
插件的单文件开关本身就重度依赖 frontmatter（`obsidian-auto-headings` 键），两者一起测才能确认不会
互相踩坑，不能只在无 frontmatter 的干净文件上验收。

### 3.16 原生风格 TOC burn-in（M10）

> **状态：规划中，构思阶段（尚未拆解到可估工时的程度）。** 目标：在单文件内提供一个"真正写入
> Markdown 源文本、且持续与标题结构同步"的目录块——不同于 Dataview（渲染层，依赖额外插件、只有装了
> 才能看）与 Table of Contents 插件（一次性插入静态文本，改了标题后不会自动更新），本功能与本插件的
> 编号本身共享同一条"burn-in 真实文本"哲学：目录内容离线可读、随文件导出/复制，不依赖任何其他插件
> 就能工作。见 [2.2 非目标](#22-非目标)的翻案记录——"生成目录"曾在同一开发周期内先被排除、又基于
> "这不是重复造 Dataview/Table of Contents 的轮子，而是做它们都做不到的事"这个论点被重新纳入，取舍
> 过程见 `doc/log.md` 本周期记录。

**触发方式：** 专属围栏代码块（info string 为 `toc`），用户在文件任意位置手写一个空的 `toc` 代码块，
插件识别并在其中写入生成的目录文本。围栏是否需要携带参数（如收录到第几级）见下方"层级折叠"一节，
具体语法留待详细设计定案。

**更新时机：** 复用现有编号引擎完全相同的防抖触发路径（`scheduleRenumber`，见
[3.9](#39-触发与防抖)），不新增一套独立的监听/防抖机制。每次防抖到期后的整文件重排里，在完成标题
编号之后、写回事务之前，额外生成/刷新 `toc` 块的内容，与编号改动合并进同一个 `editor.transaction`
一次性写回（沿用 [3.3](#33-写入文件模型)"单一事务"的既有模式）。

**关键约束：不能扰动用户当前的视图位置。** 这是本功能与既有编号引擎最大的行为差异，需要显式设计
应对：
- 现有编号引擎的核心不变量是"整文件重写但**永不增删行**"（见 [3.3](#33-写入文件模型)、
  [§4 架构设计](#4-架构设计)），`backlinks.ts` 的改名配对逻辑也依赖这条不变量按行号配对新旧标题
  （源码注释原话："编号永不增删行，故按 lineIndex 配对旧/新标题即可"）。**TOC 块内容的行数会随标题
  增删/改名而变化**——这是第一个会打破该不变量的写入路径，需要与 [3.15](#315-拖放重排与结构编辑m8b)
  的"结构性剪切—拼接"写入路径同等对待：显式评估，不能默认沿用现有的"逐行 diff"实现。
- **技术方向（待验证，非定案）：** `applyRenumber` 现有的写回方式是把逐行 diff 结果收集成
  `EditorChange[]` 交给单一 `editor.transaction({changes})`（`main.ts`）；CodeMirror 6 的事务本身具备
  "变更前后自动映射光标/选区位置"的能力，只要 TOC 块的整块替换也表达成一个 `{from, to, text}` 形式的
  change（而不是对全文拼接字符串后整体替换），理论上编辑器能自动算出光标应落在变更后的什么位置，不需
  要插件手动记录/恢复光标——这与现有编号引擎"不显式操作光标、靠单一事务自然保持光标位置"是同一机制的
  延伸，而非另起一套。**但**这个假设目前只在"内容不增删行"的场景下被现有测试隐式验证过，TOC 块这种
  会改变行数的场景是全新的，必须补专门验证（含垂直滚动位置是否也随之正确保持——这块 CM6 事务的位置
  映射不一定天然覆盖，可能需要额外记录/恢复滚动锚点）。
- **落地前置**：正因为该风险与 [3.15](#315-拖放重排与结构编辑m8b) 已识别的"新写入路径"风险同源，
  建议 M10 排期不早于 M8b 的纯函数层测试基建到位——两者都需要"允许增删行的整文件重写"这条新地基，
  没必要各自单独发明一遍。

**层级折叠：** 复用 [3.14 侧栏大纲导航（M8a）](#314-侧栏大纲导航m8a)"动态层级滑块"的判定逻辑——按
当前文件实际存在的标题层级动态启用/禁用可选项，不允许选择文件里不存在的层级。与 M8a 的差异：M8a 的
滑块是运行时交互式 UI（侧栏面板内），本功能生成的是**burn 进文件的静态文本**，没有运行时可交互的
渲染层可供"折叠/展开"。因此这里的"层级折叠"实际含义是**生成时收录到第几级（H1–H6）由用户显式指定**
（即写入前手动确定"列到 Hx"），而非阅读时可展开收起——具体交互形式（围栏参数手填 / 块内一次性小控件
写回参数）留待详细设计，但**判定"某层级是否可选"这段逻辑必须与 M8a 共享同一个纯函数**，不允许两处
各写一套、行为不一致（参照 [3.14](#314-侧栏大纲导航m8a)对 Quiet Outline"面板与核心解析器各写一套"
覆辙的告诫，同一教训适用于这里）。

**未决问题（详细设计阶段需要拍板）：**
1. 围栏语法细节：`toc` 块是否需要携带参数（如收录层级上限），还是通过其他机制配置。
2. 生成的目录条目指向标题的链接形式——应直接复用 [3.12 Backlink 同步](#312-backlink-同步m710-发布前置)
   里 `backlinks.ts` 的 `displayAnchor`（生成能正确解析到含 WJ 标题的链接），不要另起一套锚点计算。
3. 一个文件内允许几个 `toc` 块（只允许一个，还是允许多个各自独立配置层级上限）。
4. 白名单豁免的标题是否应该出现在目录里——是否要与编号计数器的"豁免且不占槽位"保持一致，待决。

> **触发的性能边界（设计澄清）。** 插件**只**在「当前活动文件的 `editor-change`」「手动命令」「设置面板改配置后对活动文件即时重排」三个时机触发，且每次只解析 / 改写**当前这一个文件**；**从不**为「应用了某模板的所有文件」做全库扫描。因此**没有打开、没有编辑的文件不会触发任何插件工作**，模板再多也不会带来后台开销——路径规则只在某文件被实际编辑 / 命令时按其路径解析一次（`getTemplateForFile`）。这与「整文件重写、从不批量扫库」的核心写入模型一致（见 [3.3](#33-写入文件模型)）。

**存储分层：**

| 存储位置 | 内容 | 读写方式 |
|----------|------|----------|
| `data.json` | 路径规则（M5）、防抖延迟等插件设置（**不含白名单**——白名单随模板存于 `templates/*.json`） | `plugin.loadData()` / `plugin.saveData()` |
| `templates/*.json` | 各模板定义文件，每个模板一个文件 | `app.vault.adapter` 文件 API（`read` / `write` / `exists` / `mkdir`） |

### A.1 定位拷问：编号是卖点，还是获客诱饵？

本文 §3.12 末尾其实已经自己招供了："把插件从'标题编号'赛道里再分出'标题改名不断链'这个更少竞争
的独立价值点，**编号本身降级为可选进阶功能**"。这句话是整份规格里战略含金量最高的一句，但它的
落点曾是 M9 候选清单的倒数第二条（CR-18）。

拷问三连：

1. **谁真的需要 burn-in 编号？** Zettelkasten/原子笔记用户根本不编号；日常记录用户用大纲面板就
   够了；真正需要编号写进正文的是写规格、论文、法律文书、要导出交付的人。而这批人的下游
   （Pandoc、PDF、粘贴到 Word/知乎）正是 本文 §2.6 里 WJ 风险最集中的地方。**核心用户画像和核心
   技术风险是同一群人**——此前 spec 没有任何一处把两件事放在一起看。
2. **"改标题不断链"的受众有多大？** 所有用 `[[file#heading]]` 的人——比编号用户大一个数量级，
   且 spec 自己承认这是"社区呼声第一的痛点、仅 Header Enhancer 解决"（Header Enhancer 还要求你
   开它的编号才修链）。CR-18 做完后，本插件是唯一一个"什么都不开、只装上就不断链"的方案。
   **这才是 infra 位。**
3. **manifest description 的第一句是什么？** M9 里躺着"backlink 链接同步前置为第一句"的候选——
   内部已有共识的零风险纯文案，不该继续躺在候选池里（注意：manifest 属产物，按上架后策略须随
   下一个行为版本一并 bump，本轮纯文档升级不动它）。

**判定**：叙事倒转为"链接永不断（零配置）→ 顺便有全社区最强的编号系统"。CR-18 升格为 M12 首项。

### A.2 WJ 哨兵拷问：护城河还是数据负债？

双哨兵自愈（本文 §2.5）在"插件自身正确性"维度上是漂亮的工程——方案 A + 双哨兵把 U1/U2/U3 整类
蚕食 bug 根治了。但换到生态维度，要直面四个此前未直面的问题：

1. **§2.6 列的是"条"，真实的风险是一个"类"。** 任何对标题文本做精确匹配的下游都会被打穿：
   Dataview 已披露，但 obsidian-tasks 的 heading 过滤、QuickAdd/Templater 用户脚本里的标题匹配、
   Various Complements 的补全、乃至 Obsidian Publish 生成的 URL 锚点（WJ 会被 percent-encode 成
   `%E2%81%A0` 进 URL）——都是同一类伤。逐插件打补丁式披露不可持续，应升级为**一次性把契约说清、
   任何下游都能照办**的《标记字符契约》（→ `doc/marker-contract.md`，英文，因为受众是下游开发者）。
2. **WJ 没有命名空间。** 参考来源 gurjar1/auto-heading-obsidian 也用 U+2060。两插件共存（或用户
   从它迁移）时，对方写的 WJ 可能被本插件的 `stripPrefix` 认作自家边界——低概率，但一旦发生就是
   "插件互相吃对方编号"的诡异现象，且用户完全无法诊断。至少登记 testplan 共存场景 + README 提示
   互斥。
3. **剪贴板风险的正确解法不是"披露"，是"拦截"。** README 现在教用户手动 `.replace(/⁠/g,"")`——
   这是把成本推给用户。插件可以监听编辑器 copy 事件、对 clipboardData 做剥 WJ 净化（渲染层不变、
   文件不变、只净化出口）。一个"复制时自动清洗不可见字符"的开关，能把 §2.6 前两条风险从"已知
   风险"直接变成"已消除"，成本远低于逐平台实测微信/知乎/Notion。→ M11 信任包。
4. **卸载叙事缺位。** "清除全库编号"功能存在，但 README 里没有一节叫"如何干净地离开"。对一个往
   用户每个标题里写两个不可见字符的插件，**可退出性就是信任**。何况 H8（清库竞态）还挂着
   backlog——离场通道本身还有个未修的数据安全隐患。→ README 新节 + H8 进 M11。

**判定**：WJ 是正确的取舍（不用它就回到 E5 误伤地狱，见 §A.8 的系统论证），但配套义务要补齐：
契约、净化、配方、卸载指南。这四件事加起来的工时可能不如 M8a 的一个滑块，却直接决定"敢不敢在
重要库里装"。

### A.3 触发面拷问：插件看不见哪些写入？

`main.ts` 只挂四个事件：`editor-change` / `file-open` / `rename` / `delete`。**没有
`vault.on("modify")`**。由此推出的盲区：

1. **外部写入不可见**：Obsidian Sync / obsidian-git pull / Templater 批量建档——这些文件在被人工
   打开编辑之前不会被编号。更微妙的是：文件已打开时被 Sync 改写，`headingSnapshots` 基线不刷新
   （只在 file-open 播种、插件写回后刷新），下一次触发时快照改名表算的是"陈旧基线 → 现内容"，
   Backlink 同步可能把同步进来的改动也当成本地改名处理。多设备 + Sync 是 Obsidian 主流形态，
   这条该进 testplan（→ O2）而不是等用户报障。
2. **批量重编号缺席**：M9 有 UX 定案（路径规则行内入口 + 确认框）但没排期。没有它，"给某文件夹
   换模板"之后旧文件全是冻结的旧编号，用户感知就是"插件不生效"。这是把"配置层改动绝不批量扫库"
   的正确原则，误伤成了"用户连主动批量的入口都没有"。→ M12。
3. **文件夹级关闭成本高**："不编号伪模板"同样有定案没排期。现状逐文件写 frontmatter `false`，
   对"我只想让 `daily/` 不编号"的用户是劝退级体验。→ M12。
4. **Canvas 盲区（spec 全文零提及）**：核对 `syncBacklinks`——它遍历 `getBacklinksForFile` 返回的
   所有引用方、只按 `TFile` 收窄、**不过滤扩展名**，然后对内容跑 `rewriteBacklinksInContent` 的
   `[[…]]` 正则。Canvas 文本节点里的 `[[note#锚点]]` 恰好可能被正则命中并改写（在 JSON 字符串里
   做纯文本替换，碰巧无害但纯属运气），而 canvas 的 `"file"` 字段式引用则永远匹配不到、静默断链。
   这属于"未设计、行为靠巧合"的区域：要么显式跳过 `.canvas`，要么显式支持并测试。→ M11 拍板项
   + testplan O1。

### A.4 Backlink 同步拷问：最接近 infra 的资产，也是最大的信任敞口

这套实现（改名表按行配对、快照基线、`foldSelfBacklinks` 同事务折叠、`vault.process` 原子写、
重复锚点保守不改）是全插件质量最高的部分，比 Header Enhancer 的四处改进都真实成立。拷问点恰恰
因为它重要：

1. **默认开 + 静默改写他人文件 + 不在撤销历史内**，三件事叠在一起是插件最危险的行为组合。护栏
   （首次 Notice、保守跳过、静默降级）都有，但**审阅模式（决策 D2）曾在 M9 候选池里排在大纲面板
   之后**——一个改写你二十个文件的功能，其"先看清单再批准"模式的优先级，低于一个侧栏树？这是
   本次审查发现的最明显的优先级倒挂。Git diff 噪音（§2.6）和批量误改焦虑，审阅模式一并解决。
   → M11 首项。
2. **H8（清全库用 `vault.read`+`modify` 的竞态）**：`foldSelfBacklinks` 的修复注释里已经把同源
   竞态的机理写得清清楚楚，同一课却没回头补到 `clearAllVaultNumbering` 上。"卸载通道上挂着已知
   竞态"不该是 backlog。→ M11（与"清库支持撤销"同段代码，一并修）。
3. **M12 场景（大库多引用文件性能）🔲 手验未做**：改名一个被 50 个文件引用的标题，50 次串行
   `vault.process` 的体感如何，1.0 上架后仍未知。infra 级插件的第一要求是"永远不让人觉得卡"。
   → M11。
4. `getBacklinksForFile` 是半公开 API，已有降级——处理是对的；商店人工审核可能问，关于页/README
   可主动说明降级行为。

### A.5 导出拷问：调研项的姿态太被动

对一般插件，"带编号导出"标为调研项是合理保守；对**以 burn-in 为哲学**的插件，导出就是 burn-in
的终点站，不能永远待验证：

- **Pandoc 双重编号**：用户拿编好号的文档跑 `pandoc --number-sections` 会得到 `1 1 Introduction`。
  这不是 WJ 问题，是 burn-in 模式的**固有语义冲突**，一段"导出工作流建议"（导出前跑清除命令 /
  提供十行 Lua filter 剥 WJ + 编号）就能解决，零插件代码。→ README「导出与外发」骨架本轮已建。
- **核心 PDF 导出 / Publish**：WJ 在导出 PDF 的字体栈里是否 tofu、Publish 锚点 URL 长什么样——
  两个验证各需十分钟一个真实 vault，拖着不验的成本是 README 里那句"已知风险、不承诺具体表现"
  持续侵蚀信任。→ M11 导出验证矩阵。
- 让"验证矩阵"成为卖点而非欠条：README 新节先立骨架（状态如实标"待实测"），矩阵结论逐项回填。

### A.6 Milestone 排序拷问：M8a 值得当下一站吗？

旧排序：M8a 大纲面板 → M8b 拖放重排 → M9 候选池 → M10 TOC。逐个拷问：

- **M8a**：大纲面板是红海（Quiet Outline、Modern Outline、核心 Outline）。调研做得扎实（避开 QO
  的跳级 bug、彩虹色、iCloud 冲突），但"比 QO 好一点的面板"不构成装机理由——面板只对已经用你编号
  的人有增量。**它是留存功能，不是获客功能。**
- **M8b**：spec 自己承认"风险高于以往任何一个 Milestone"。在审阅模式、H8、批量重编号这些信任债
  未清时先开高风险结构编辑，顺序反了。
- **M10（TOC burn-in）**：被低估——"目录随标题持续同步的真文本"确实无人做到，与核心哲学同构，
  且依赖 M8b 基础设施这一点已被正确识别。维持在 M8b 之后。
- **M9 候选池里睡着的，恰恰是 infra 化的全部原料**：CR-18、审阅模式、批量重编号、伪模板、
  注释块解析、全库断链修复。

**重排结论（已落 本文 §5）**：M11 信任包 → M12 独立价值包 → M8a → M8b → M10；M9 保留为候选池，
升格条目移出。

### A.7 infra 级插件的路径：从"功能全"到"被依赖"

"infra 级"的定义不是功能多，是**其他插件和工作流开始依赖你的存在**。差距清单：

1. **对外 API 面 = 0**。现在所有能力都锁在命令和设置里。infra 化的第一步是把"标题改名事件"广播
   出去（自定义 workspace event 或挂在 plugin 实例上的公开方法 + 类型声明）：让 Tasks、Dataview
   脚本、别的编号/大纲插件能消费"某文件第 N 行标题从 X 变成 Y"。本插件已有全社区最可靠的改名
   检测引擎（快照基线），**它作为服务的价值大于作为内部实现**。→ M12。
2. **迁移即增长**。Number Headings 有 8.5 万下载、停更约 2.5 年、38+ issue 堆积——而"清理非本插件
   的标题编号"命令**天然就是它的迁移工具**，README 里却只把它描述成通用功能。写一篇《从 Number
   Headings 迁移》指南（一条命令接管 + 功能对照），发到论坛/Reddit，是当前性价比最高的增长动作，
   零代码。→ README 新节（本轮）+ M12 社区发布项。
3. **开发者文档语言**。仓库纪律是全中文——作为个人仓库纪律无可指摘，但 infra 级插件需要外部贡献者
   与下游插件作者。不必翻译全部，但《标记字符契约》、公开 API 文档这两份必须英文（契约本轮已立）。
4. **可退出性与稳定性承诺**：卸载指南（本轮 README 新节）、frontmatter 键 `obsidian-auto-headings`
   永不变更的承诺（0.7.26 的决策写成对外承诺，进契约文档）、性能预算（多大的文件/多少引用内保证
   不卡，M11 实测后立数字）。
5. **已经做对的**：上架后"仅行为变化才 bump"的版本纪律、CI 兜底、release 自动化、半公开 API
   降级、testplan 152/165 绿——这些是 infra 的地基，不用动。

### A.8 专题：WJ 能否被 CodeMirror 原子区域替代？

> 用户提问（2026-07-10）：能否把编号做成 CM6 原子区域——用户无法直接修改、不引入 WJ 污染、
> 纯文本依然天然，只是被"接管"一部分区域？

#### A.8.1 先把 WJ 的职能拆开

WJ 在现方案里承担**两个完全不同的职能**，讨论替代必须分开：

| 职能 | 内容 | 依赖什么 |
|------|------|----------|
| **身份**（identity） | 在字节层面标记"这段是插件写的"——使 `stripPrefix` 幂等、区分手写伪编号（`## 1. 引言`）、让"清除自家编号"与"清理外来编号"两条命令能分家、识别降级残留 | 必须**与文件字节同生共死**：跨会话、跨设备、跨编辑器、跨 sync/git 都不丢 |
| **完整性**（integrity） | 防御用户编辑破坏前缀边界——双哨兵自愈（0.7.20）整个是为它 | 只需要在**编辑发生的地方**起作用 |

#### A.8.2 CM6 原子区域的能力边界

Obsidian 桌面与移动端均为 CodeMirror 6，插件可经 `registerEditorExtension` 注入扩展。原子区域
（`EditorView.atomicRanges` facet + 装饰 RangeSet）能提供：

- **编辑器内的交互屏障**：光标移动跳过区域、无法在内部落点、Backspace/Delete 把区域当作单元；
  配合 `changeFilter`/`transactionFilter` 可彻底拦截落在区域内的修改。
- **位置随文档变更自动映射**：`RangeSet.map` 让装饰跟着编辑走，会话内位置追踪是免费的。

它**不能**提供：

- **任何持久性**。装饰是视图状态，文件关闭即消失；重开文件必须重新回答"哪些区域是编号"——
  这一步要么靠字节内标记（回到 WJ），要么靠模式匹配，要么靠外部位置存档。**原子区域本身回答
  不了身份问题，它只消费身份。**
- **编辑器之外的覆盖**：未打开文件的清除/批量命令、Backlink 改写他人文件（`vault.process`
  无编辑器）、外部编辑器 / Sync / git 的写入、阅读模式——全部在原子区域的射程之外。

#### A.8.3 三条"去 WJ"路线逐一检验

**路线一：纯模式匹配（无任何标记）。** 信息论上不可行——`## 1. 引言`（手写，必须保留）与
`## 1. 引言`（插件写，必须可重排）**字节完全相同**，无标记则原理上不可区分。历史已为此付过学费：
0.6.6 之前正是模式匹配，产出 U1/U2/U3 整类蚕食 bug（`API 设计` 被 upper-alpha 样式当序号吃掉、
`2024 总结` 被吞、非幂等逐次蚕食）。方案 A（纯 WJ 边界）就是从这个坑里爬出来的。**枪毙。**

**路线二：位置 sidecar（data.json / 边车文件存每文件的前缀 span）。** 身份与内容分离存放，
意味着任何一次外部写入、同步时序错位、git checkout 都可能造成**元数据与内容漂移**；漂移的后果是
按错位置剥离 = **静默吃正文**，比 WJ 的最坏情况（外部搜索不命中）严重一个数量级。且 sidecar 自身
有云同步冲突问题（Quiet Outline #308 的 iCloud 前车之鉴，本文 §3.14 已引为反面教材）。数据主权上
反而更差：文件"看似干净"，但离开插件（或 sidecar 丢失）后满是**无法识别**的死编号——连"清理"都
无从下手。对比之下 WJ 有个被低估的优点：**卸载多年后**，重装插件（或一行 sed）仍能把自家编号
精确剥净，因为身份活在字节里。**枪毙。**

**路线三：原子区域 + 会话内位置追踪。** 会话内确实可行：插件写回时自己知道前缀区间，装饰随编辑
映射，"装饰集里的就是我的前缀"在会话内成立。但**会话边界即失效**——重开文件、重启 Obsidian、
换设备，都回到"重新发现"问题，仍需字节内身份。它无法独立成立，只能作为叠加层。**不能替代，
可以叠加。**

#### A.8.4 结论：不能替代，但应当叠加

**原子区域完美对口的是"完整性"职能，完全对不上"身份"职能。** 而双哨兵（0.7.20）恰恰是为完整性
打的补丁——所以正确的组合是：

> **WJ 继续当身份（文件格式不变、契约稳定），原子区域进编辑器当完整性的第一道防线。**
> 防护栈从三层变四层：**原子区域（编辑器内让破坏根本不发生）→ 方案 A（无 WJ 不剥）→
> 双哨兵自愈（外部编辑损伤的兜底）→ 清除命令（最后兜底）。**

附带收益：

- 编号在交互上"感觉是托管的"（光标跳过、删不动、整体选中）——恰好向用户传达"这不是你的正文"的
  心智模型，对信任叙事有利；顺带消灭 IME 在前缀边界的组合事故面（J8 的邻近场景）。
- WJ 反而让原子区域的定位既 O(n) 又零误判：扫描文档找 WJ 对即可标注区间，无需模式猜测——
  **两个方案是互相成就，不是互相替代。**

需要拍板的交互细节（登记 testplan O6，实现属 M11）：

- 整行/整段删除必须放行（原子区域随整段删除天然允许，需测试确认）；
- 插件自家写回（`applyRenumber` 的 `editor.transaction`）需豁免拦截（模块级标志或事务注解）；
- 与 Vim 模式、编辑器内搜索替换、多光标的相互作用；
- 移动端实测（同为 CM6，预期可用，需一轮实机验证）；
- **原则修订**：本文 §2.3 的「前缀可手改是预期行为」退役 → 改为「编辑器内前缀只读；编辑器外
  改动由双哨兵自愈兜底」。设置内给开关（保守起见默认关一个版本，实测稳定后翻默认）。

#### A.8.5 真正的"零 WJ"路径：虚拟编号模式（opt-in 候选）

如果目标是"文件里一个 WJ 都没有"，诚实的路径不是"烧入但无标记"（§A.8.3 已证不可行），而是**不烧入**：
编号只作为 CM6 行内 widget（编辑视图）+ `MarkdownPostProcessor`（阅读视图）渲染，**永不写入文件**。

| 维度 | 虚拟模式 | burn-in 模式（现状） |
|------|----------|---------------------|
| 文件纯净度 | ✅ 零改动、零 WJ | 前缀 + 双 WJ 写入 |
| Backlink | ✅ 标题文本永不变 → **编号引发的断链问题整体消失** | 需要整套同步机制 |
| 阅读模式 | ✅ 天然支持（postprocessor） | ❌ 现状不支持（M9 旧候选） |
| Obsidian 内 PDF 导出 | 预期出现（走阅读视图渲染管线，**待实测**） | 编号在，但 WJ 风险待实测 |
| 外部编辑器 / GitHub / Pandoc | ❌ 编号不可见 | ✅ 编号可见（burn-in 的存在理由） |
| 外部精确检索 / Dataview | ✅ 无 WJ 干扰（编号本身也搜不到） | 受 WJ 影响（契约 + 净化缓解） |

它不替代 burn-in，是**与 burn-in 并列的第二哲学**：本文 §2.2 的非目标写的是"仅注入渲染层"不做
——作为**默认哲学**保留；虚拟模式是显式 opt-in（模板级或全局模式切换，同一文件二选一），面向
"生态纯净优先"的人群。把插件从"一种取舍"升级为"两种取舍都提供"，是 infra 级的姿态（参考 TOC
非目标在 0.7.28 的翻案先例，翻案理由同样是"与哲学的关系想清楚了"）。

代价与约束：双模式 = 双倍状态空间；切换语义（虚拟→burn-in = 一次编号写入；burn-in→虚拟 = 先清除
再只渲染）必须定义清楚；排期**不得早于 M11 信任包**（先把现有模式的信任债清完）。→ 进 M9 候选池，
吸收合并旧候选"阅读模式下插件运行"（虚拟模式天然覆盖之）。

#### A.8.6 顺带检验：能否退回单哨兵（少写一个 WJ）？

原子区域上线后编辑器内损伤面趋零，尾哨兵被误删的高发场景（改后缀）大幅收窄——那尾哨兵还要吗？
**要。** 外部编辑器损伤仍在射程外；双哨兵自愈已实现且经 E14–E18 验证；更重要的是**文件格式稳定
本身就是契约的一部分**（marker-contract 对下游承诺格式不漂移），为省一个不可见字符而变更格式、
触发一轮存量迁移，得不偿失。**维持双哨兵。**
