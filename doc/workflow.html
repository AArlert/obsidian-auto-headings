Design Principles

三条设计哲学

工作流的一切细节都从这三条推导出来。

🗄️

厚存储 · 薄读口

历史细节全部留档（log-archive、status-archive、spec），但接手时的热读取路径被压缩到约百行——status 首行 + log 最新块 + testplan 待办摘要。

🤝

机械交脚本 · 语义留 Agent

归档、滚动、计数、校验、版本号同步都是幂等脚本（docs.mjs / bump.mjs）。Agent 只写两样东西：新周期块 + 一句话状态概括。

📌

单一事实源 + 守卫

同一份设计只详写一处（规格→spec、验证→testplan、周期→log），其余只放链接。写了两处的，用自动比对守卫逼着保持一致。

Memory System

记忆系统：接手只要一条命令

新 Agent 接手时不通读代码、不翻全部日志——跑 npm run docs -- --handover，三段信息一次打印：

status.jsonl · 首行

当前总览

版本、里程碑、一句话现状、下一步。首行强制精简（超 200 字告警），细节一律下沉 log.md。

log.md · 最新周期块

开工起点

上个周期「做了什么 / 没做什么 / 下一步 / 验证方式」。只保留最新 3 块，更旧自动滚入归档。

testplan.md · §2 摘要

待办清单

场景真值表按状态计数，列出全部非 ✅ 行（ID + 行号）。只统计场景区间，说明行不误报。

再往深处走才按需 grep 定位：查规格先取 spec.md 节行号再截读；查历史周期在归档文件里搜版本号。✅ 的场景、已归档的周期，不读。

Anti-Drift Guards

四道防漂移守卫

文档规则靠自觉必然漂移。这里的每一条纪律背后都有一个机器执行的守卫。

pre-commit · --check📚

日志膨胀守卫

log.md 周期块 > 3 个 = 写了新块忘归档，提交被拦下。

pre-commit · --check📇

状态索引守卫

status.jsonl 概括行 > 12 条即拦下；每行超 200 字告警——索引永远保持一屏内。

pre-commit · --check🌲

目录树守卫

log.md 目录结构块与磁盘 .ts/.mjs 双向比对——新增 / 拆分文件忘回填地图，立即报错。

push / PR · CI🛡️

CI 机器兜底

--no-verify 骗得过钩子骗不过 CI：同一套守卫 + test + lint + build 在云端再跑一遍。

Automation

脚本一览

每个脚本都幂等、可重复跑，「写」与「挪」解耦。

🧭

docs.mjs

--handover 接手三段式；缺省归档 log / 滚动 status / testplan 摘要；--check 守卫模式给钩子和 CI 用。

🔢

bump.mjs

版本号唯一真相源，一条命令同步五处文件；支持打磨递增 / 新里程碑 / 显式指定。

📦

sync-release.mjs

构建产物刷进 release/——每个周期必须产出可实测插件并随提交入库。

🚀

sync-plugin-repo.mjs

开发（monorepo）与发布（干净插件仓库）分离，向 Obsidian 社区目录发布时单向同步。
