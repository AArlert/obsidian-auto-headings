# Harness 工作流思想提炼：面向 IC 设计验证 Agent 工作的迁移设计

> 本文档提炼自本仓库（`obsidian-auto-headings`）自身的多 Agent 协作 Harness 机制——
> `CLAUDE.md` 交接协议、`doc/log.md` + `doc/status.jsonl` 分层记忆、`scripts/docs.mjs`
> 机械脚本化、pre-commit / CI 两级门禁，以及 `doc/testplan.md` §4 已经借用硬件验证 UVM
> 思想设计的约束随机测试框架。目的是把这套协作纪律迁移到 **IC 设计验证 Agent 工作**场景，
> 供外部项目参考落地，不属于本插件自身的规格或验证设计，仅作知识沉淀保留于此。

## 0. 一句话总纲

Harness 的本质是：把"事实"和"整理事实的动作"分离——语义内容（发生了什么、下一步做
什么、约束是什么）由 Agent 手写并保持精简；机械动作（归档、滚动、校验、版本同步）交给
幂等脚本；Agent 之间只靠一份可被单命令摘要的"交接记忆"传递上下文，绝不要求下一个
Agent 通读全部历史。

## 1. 核心原则（可直接当公理表抄）

| # | 原则 | 本仓库体现 | 为什么重要 |
|---|------|-----------|-----------|
| P1 | **单一事实源（Single Source of Truth）** | 规格只写在 `spec.md`，验证设计只写在 `testplan.md`，周期细节只写在 `log.md`；其余文件只放一行概括+链接 | 防止多处复制导致的漂移；改一处即全局生效 |
| P2 | **交接记忆分层，读盘按需** | `status.jsonl` 首行总览 → `log.md` 最新块 → `spec.md`/`testplan.md` 按 grep 定位 → archive 文件只在追溯历史时才翻 | 长期项目里"完整上下文"会无限增长；分层保证接手成本恒定，不随项目历史线性增长 |
| P3 | **机械与语义解耦** | Agent 只写"新周期块"和"新场景行"；`scripts/docs.mjs` 负责归档、滚动、校验、打印摘要，且**幂等、可重复跑** | Agent 的 token/精力该花在判断上，不该花在数组腾挪、格式对齐这类确定性劳动上 |
| P4 | **软门禁本地、硬门禁远程** | pre-commit（可 `--no-verify` 跳过）做文档纪律检查；CI（不可跳过）做完整质量门槛 | 本地允许效率优先的例外，但必须有一道"机器兜底"防止例外扩散成常态 |
| P5 | **状态转移是一等公民，不是单次变换** | testplan §1："凡是配置字段可被用户改动，都要测状态转移"，而非只测"干净输入→一次输出" | 现实世界的 bug 绝大多数出在"旧状态 + 新操作"的组合上，穷举单次输入根本测不到 |
| P6 | **约束是"当前已知正确行为的精确刻画"，会随修复而系统性放松** | testplan §4："每条约束对应一个已登记 bug/取舍"；修好 bug 就放开对应约束让框架自动覆盖更大空间 | 把"我们目前能保证什么"显式化，且用一个单向棘轮机制（只放松、不收紧）持续逼近"覆盖用户全部操作" |
| P7 | **多记分板互补而非单一断言** | 参考记分板（真值比对）+ 幂等性记分板（`f(f(x))===f(x)`）+ 领域不变量记分板（S4/S5/S6/S7）并存 | 不同记分板逮不同类型的 bug；单一断言必然有盲区 |
| P8 | **显式登记"故意不测什么"，而不是任由覆盖面自然衰减** | testplan §4.1.1 表格末尾"🚫 不入 UVM"一列，并说明留给谁（`main.test`/`user_tests`） | 未覆盖分两种："忘了"和"评估过决定不做"，必须能区分，否则每次交接都要重新论证一遍 |
| P9 | **已知边界要有测试钉住，而不是留成隐患** | 如"从未接触过"判定是整文件级、部分接管边界——专门写了回归测试固化这个边界，log.md 里注明"不是遗漏" | 防止后人（或下一个 Agent）把"刻意的取舍"误判成 bug 去"修复"，破坏设计 |
| P10 | **产物随源码一起入库，不依赖构建步骤才能验证** | `release/` 目录连同构建产物一起提交 | 缩短"改动→可验证"的回路，尤其利于人工/外部下游即时验证 |

## 2. 分层记忆架构

```
接手第一条命令 ──▶ 单命令摘要（只读、聚合三处）
                     │
       ┌─────────────┼─────────────────┐
       ▼             ▼                 ▼
 status.jsonl     log.md（最新块）   testplan 摘要（按脚本生成，非整读）
 首行=当前总览     "做了什么/没做什么/     ✅/❌/⚠️/🔲 计数 + 非✅行清单
 其下=历次概括     下一步/验证方式"
 （滚动保留N条）    （滚动保留N块）
       │             │
       ▼             ▼
 status-archive   log-archive.md
 .jsonl（旧概括）  （旧周期块，倒序，仅追溯用）
```

**关键设计点**：

- `status.jsonl` 首行是**唯一被反复读的一行**，因此有硬性字数上限（本仓库 200 字软限），
  细节一律下沉到 `log.md`。
- `log.md` 顶部有"常青块"（强制规则、目录结构约定）——不受滚动归档影响，永远可见，
  因为这是每个 Agent 都必须先知道的行为约束。
- 归档是**单向、只增不减**的：旧块搬到 archive 文件，绝不删除，保证可追溯性与
  "可回滚记忆"零信息损失。

## 3. 机械脚本化设计（`scripts/docs.mjs` 的可迁移模式）

这个脚本做的五件事，抽象出来是一个通用模式：

1. **归档**：按"块是否带日期标题"区分"周期块"（受归档管控）与"常青块"（永远保留），
   只保留最新 N 个周期块。
2. **滚动**：索引类文件（jsonl）首行永久保留，其余按数量滚动到 archive。
3. **摘要**：对"真值表类"文档（testplan）只统计特定章节区间的状态标记，生成"非通过项
   清单"，而不是把整份文档打印出来。
4. **校验（`--check` 模式）**：不修改任何文件，只检查是否超限/格式是否合法，非零退出——
   这是 pre-commit/CI 复用的模式。
5. **目录树漂移守卫**：把文档里登记的"文件清单"与磁盘实际文件做**双向 diff**（漏登记 vs
   幽灵登记），防止"新增文件忘更新地图"这种最常见的文档腐烂。
6. **接手模式（`--handover`，只读）**：一条命令聚合打印"该看的三处"，取代人工分别打开
   三个文件。

**设计纪律**：脚本必须**幂等**（重复跑等于跑一次，不会因多跑一次而损坏状态），这样
Agent 不需要在"我是不是已经跑过归档了"这种状态判断上耗费判断力，随时可以放心调用。

## 4. 质量门禁的两级模型

| 层级 | 触发点 | 可否跳过 | 检查内容 |
|------|--------|---------|---------|
| **本地软门禁** | `git commit`（pre-commit hook） | 可 `--no-verify` 跳过 | 只查"文档纪律"这类快速、确定性的机械检查（周期块超限/索引超限/目录树漂移） |
| **远程硬门禁** | push / PR（CI） | 不可跳过 | 完整质量门槛：文档守卫 + 全部测试（含随机压测）+ lint + 格式 + 构建 |

**哲学**：本地允许"先跑起来，稍后补票"的效率优先例外，但必须存在一道谁都绕不过的机器
闸门，否则"这次先跳过"会系统性累积成债务。

## 5. 验证方法论内核（本仓库已借用 UVM 思想，可直接对齐 IC 验证领域）

这是最值得直接拿走的部分——因为这个 Markdown 插件的测试层，已经在用硬件验证的语言在
设计自己：

### 5.1 状态转移优先于单次变换

不测"干净输入→一次输出"，而是把**操作序列**当作一等验证对象：
`裸状态 → 触发 → 状态A → 改配置 → 触发 → 状态B → ...`。多数深层 bug 只在"旧状态的
残留 + 新配置的解读方式"组合下才暴露。

### 5.2 约束随机 + 参考模型（golden model）

`tests/dev_tests/uvm/` 目录下有一个明确的"World/Vault"模型——一个独立维护的参考状态
机，与真实 DUT（编号引擎）锁步运行，每步比对。这就是标准的 **reference model
scoreboard**。

### 5.3 记分板互补，而非单一断言

- **参考记分板**：真值比对（`join(rendered) === 独立计算的期望值`）——默认常绿模式。
- **幂等性记分板**：与配置无关的恒成立不变量（`f(f(x))===f(x)`）——explore 模式专门
  放开约束找 bug。
- **领域不变量记分板**（S4/S5/S6/S7 等）：针对特定命令/门控/多态解析各自的恒成立律，
  每加一个新的用户操作面就配一个新不变量。

### 5.4 约束 = 当前已知行为的精确刻画，且单向放松

每条随机化约束在文档里显式标注"对应哪个已知 bug/取舍"。**修复 bug 后就放开对应约
束**，让框架自动去覆盖更大空间；如果放开后又变红，说明这次修复不彻底——这是一个自我
校验的收敛机制，而不是一次性写死的测试。

### 5.5 功能覆盖率驱动的缺口分析

维护一张"人类全部操作 × 当前 UVM 覆盖状态"的矩阵（testplan §4.1.1），按"逮 bug 的
价值"给缺口排序（§4.1.2），每个缺口配一个新不变量（§4.1.3），分阶段实现（§4.1.5：
阶段1/2/3依次落地，每阶段都有独立的压测规模验证 + "未发现引擎 bug" 的结论记录）。

### 5.6 显式的"不入随机框架"清单

明确写出哪些操作面**故意**不纳入约束随机测试（如 DOM 交互、纯时序、纯文案），并说明
这些交给什么替代手段（集成测试/手工测试）覆盖——避免"覆盖率数字好看但实际漏了一大块"
的假象。

## 6. 迁移到 IC 设计验证 Agent 工作：具体架构建议

下面是把上述六节原则映射到一个 IC 验证 Agent 工作流的具体文件/脚本/门禁设计。§5 的方
法论本来就是验证工程师的母语，不需要翻译，只需要把 §1-§4 的"协作记忆 + 自动化纪律"
套上去。

### 6.1 文档层（`doc/` 目录直接照搬角色，改内容语义）

| 原文件 | IC 验证语境下的角色 | 内容示例 |
|--------|---------------------|---------|
| `status.jsonl` | **验证状态索引** | 首行：`{version, milestone, summary: "IP-X 功能覆盖率 87%，2条已知 bug 待修", next: "补 outstanding 覆盖率 bin Y"}`；其下每行一个验证周期概括 |
| `log.md` | **验证交接日志** | 每周期："本周期加了什么 checker/激励/覆盖率 bin，发现了什么 bug（含 waveform/seed 复现方式），没做什么，下一步" |
| `spec.md` | **设计规格 + 验证计划（DVP, Design Verification Plan）** | 功能列表 + 每个功能对应的验证策略（定向用例 vs 约束随机 vs formal） |
| `testplan.md` | **测试场景真值表**（本来就是 IEEE 1800/UVM 测试计划的标准产物） | 场景 ID + 激励 + DUT + 期望 + 状态（✅/❌/⚠️/🔲）+ 关联 testbench 文件 |
| `log-archive.md` / `status-archive.jsonl` | 历史周期归档，只在"这个 bug 什么时候引入的"这类溯源时才翻 | 同 |

### 6.2 testplan.md 的验证维度建议（对齐本仓库 §4 的设计但用 IC 语言）

```
## 1. 核心理念：状态转移测试
  → 对齐："复位后连续多个 transaction 的交互"，很多 IC bug 只在
    "上一个 transaction 的残留状态 + 新 transaction 的时序"组合下才暴露
    （如 FIFO 边界、pipeline hazard、arbiter 饿死）。

## 2. 场景清单（定向用例真值表）
  | ID | 激励 | DUT/接口 | 期望 | 状态 | 关联 testbench |

## 3. 已知 bug 汇总
  | ID | 现象 | 根因 | seed 复现方式 | 状态 |

## 4. 约束随机 + Scoreboard（这部分比本仓库更专业，直接用验证工程师自己的方法论）
  - 参考模型 scoreboard（golden model 锁步比对）
  - 不变量 scoreboard（协议合规性，如 AXI ordering、cache coherence）
  - 覆盖率 bin 表 + 缺口排序
  - 约束表：每条约束 → 对应哪个已知 bug/tradeoff → 状态（已放开/仍保留原因）
  - "放开约束=扩大覆盖"的单向棘轮记录

## 5. 明确不入随机框架的部分
  - 纯时序收敛类（PLL lock、DFT scan chain）留 formal/gate-level sim
  - 纯 UI/寄存器映射类留手工 checklist
```

### 6.3 脚本层（`scripts/docs.mjs` 的等价物）

写一个 `scripts/vplan.mjs`（或复用 Python，取决于验证平台工具链），实现：

1. **归档**：`log.md` 只保留最新 N 个"带日期的验证周期块"，旧的滚入 archive。
2. **摘要**：解析 testplan 的场景表，输出"非 ✅ 行清单"——**同时可以直接解析覆盖率
   数据库导出（UCIS/URG report）**，把"覆盖率缺口 bin 列表"自动拼进摘要，而不是让
   Agent 手工读覆盖率报告。
3. **一致性守卫**：testplan 里登记的 testbench/sequence 文件名 vs 实际 `tb/`、`seq/`
   目录下的文件做双向 diff——防止"加了新 sequence 忘登记进 testplan"或"testplan 引用
   了已删除的 sequence"。
4. **回归结果同步**：如果有回归跑批系统（regression farm），可以让脚本读取最近一次
   回归的 pass/fail 摘要，自动写入 `status.jsonl` 草稿行（Agent 只需确认/补一句话概
   括，而非手工统计通过率）。
5. **`--handover`**：一条命令打印"当前覆盖率总况 + 最新一次交接的做了什么/下一步 +
   未过场景清单"。
6. **`--check`**：给 pre-commit/CI 用，纯校验、非零退出。

### 6.4 门禁层

| 层级 | 对应动作 | 检查内容 |
|------|---------|---------|
| 本地软门禁（pre-commit） | 提交 testbench/testplan 改动前 | 文档纪律（周期块/索引超限、文件登记漂移）+ **lint-only 编译**（如 `vlog -lint` / verilator `--lint-only`，秒级反馈） |
| CI 硬门禁 | push / PR | 文档守卫 + **完整/精简回归集**（smoke regression，非全量）+ 覆盖率报告生成 + lint |
| 夜间/周期性硬门禁（额外一层，IC 验证比软件多这一层） | 定时触发 | 全量约束随机回归（大 seed 数）+ 覆盖率合并 + 覆盖率闭合趋势跟踪 |

### 6.5 Agent 交接协议模板（直接抄 log.md 的块结构）

```
## 2026-07-06 <IP名/milestone> <一句话主题>（<分支名/agent标识>）

**做了什么**：
1. 新增/修改的 testbench 组件、sequence、checker、覆盖率 bin（逐条，含文件路径）
2. 发现的 bug（现象 + 根因 + 复现 seed + 修复状态）
3. 覆盖率变化（此前 X% → 现在 Y%，新增哪些 bin 被打中）

**没做的**：明确列出评估过但本周期未做的事，及原因

**验证方式**：跑了哪些回归（seed 数、时长、pass/fail）、新增不变量是否常绿

**下一步**：下一个 Agent/工程师接手应该先做什么
```

### 6.6 最小可行落地清单

1. 建 `doc/status.jsonl`、`doc/log.md`（含"强制规则"常青块）、`doc/testplan.md`
   （含 §1 状态转移理念 + §4 约束随机设计）。
2. 写 `scripts/vplan.mjs`：归档 + 摘要 + 文件登记漂移守卫 + `--handover` + `--check`。
3. 接一个 pre-commit 钩子跑 `--check`（可 `--no-verify` 跳过）。
4. CI 跑 `--check` + smoke 回归 + lint（不可跳过）；夜间跑全量约束随机回归。
5. 约定："每个验证周期结束必须在 `log.md` 顶部追加一块，并跑归档脚本"——写进 Agent
   系统提示/CLAUDE.md 等价文件里，作为强制规则常青块。
