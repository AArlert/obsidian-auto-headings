# README 改版调研报告

> 本文件是**一次性调研产出**，供决定 README 改版方向使用，不是常驻规格文档——结论一旦落地
> 到新版 `README.md`/`README.zh.md`，本文件即可删除，避免与「单一事实源」（`spec.md`）重复。
> 产出于 2026-07-04，讨论轮，未改任何源码或既有文档。

## 0. 结论先行

- 当前中英 README 的**内容密度已经不输、甚至反超两个对标对象**，短板不是"没写"，而是
  **组织方式**（时间线堆叠而非按用户旅程分层）和**呈现形式**（全篇纯文字，零图示）。
- `spec.md` 里记录的设计权衡，很多是"用户永远不会主动问，但一旦知道就会信任这个插件"的
  证据——这些权衡本身就是最有说服力的 README 素材，比空喊"稳定可靠"有用得多。
- 建议新 README 按**「基础使用」→「进阶功能」**两层组织（用户已认可此方向），细分见 §3。

## 1. spec.md 里对 README 写作有用的设计权衡

逐条标注来源章节，写 README 时可直接引用其中的"为什么"而非只写"是什么"。

### 1.1 核心哲学："只加减前缀，绝不碰你的结构"

- **插件永不改写标题的 `#` 层级**（[§2.2 非目标](spec.md#22-非目标)、[§3.4](spec.md#34-起始编号层级与多个-h1)）：
  历史版本曾把多余 H1 降级/级联下移子树，后彻底移除——因为这等于替用户重排文档结构。
  取而代之是 `topLevel`/`bottomLevel` 两个边界配置。
  **README 价值**：这是与"自动化工具会不会自作主张改我的文档"这种潜在焦虑直接对话的点，
  目前 README 只在 Features 列表第 3 条一笔带过，可以提到更前面。
- **配置不入 frontmatter**（[§2.2](spec.md#22-非目标)、[§3.2](spec.md#32-单文件开关frontmatter)）：
  模板/白名单/路径规则全部只存在设置 GUI + `templates/*.json`，frontmatter 只留一个开关键。
  **README 价值**："打开你的笔记，看不到任何插件痕迹"——对在意笔记可移植性的用户是加分项，
  目前完全没写进 README。

### 1.2 Word Joiner 双哨兵：从"能用"到"自愈"的技术故事

- 单哨兵方案（0.6.4 起，[§3.5](spec.md#35-内部计数模型)）：编号前缀末尾写入不可见 WJ（U+2060），
  剥离时只认它，从根源解决"标题正文自己以数字开头"（如 `## 2024 年度总结`）被误吃的问题
  （[§2.4 方案A](spec.md#24-保留正文起头的数字方案a)）。
- 双哨兵自愈（0.7.20，[§2.5](spec.md#25-双哨兵自愈删后缀致序号重复的根治0720)）：真实用户报告——
  删除后缀时连带删掉了紧邻正文的尾哨兵，导致下次编号在残留序号上叠加、产生 `一、一、标题2`
  这种重复。修复方式是前缀**首尾各放一个哨兵**，尾哨兵被误删时首哨兵仍存活，成为"这里确实
  有插件前缀"的结构性证据，从而安全自愈。
  **README 价值**：这是全篇最有技术含金量、也最能建立信任的细节——"连你手滑删错字符，
  它都能认出来自己写的东西并纠正"。当前 README 的"How it works"一节只讲了单哨兵，
  没提双哨兵自愈这个后续加固，可以补一句作为"久经打磨"的证据，而不需要展开原理。
  同时应保留对 gurjar1/auto-heading-obsidian 的鸣谢惯例（spec 里已注明参考来源）。

### 1.3 白名单"子树重置"背后有调研支撑

- 子树豁免块后计数器整体重置（决策 D1，0.7.10，[§3.7](spec.md#37-白名单系统)）：
  依据是对 APA/Chicago/IEEE、RFC、GB/T 7714 等规范的调研，**约 85% 的文档范式**里附录与正文
  编号断开、其后重新起算，所以不设开关、统一重置。
  **README 价值**：这是"我们不是拍脑袋定的默认行为"的例子，可以在白名单功能描述里加一句
  "基于主流引用规范调研的默认行为"，比单纯罗列三种匹配模式更有说服力。

### 1.4 `ancestorNumeral`：解决中文书 vs 提纲两种排版惯例的根本冲突

- [§3.6](spec.md#36-模板系统)："本级单独长什么样"与"本级作为祖先嵌进更深前缀里长什么样"是两件
  方向相反的事——中文书排法要求章节标题显示"一"但子节号里降级为阿拉伯 `1.1`；提纲排法
  要求祖先段保留各自字形（`1.a.①`）。单一 `numeral` 字段无法兼顾，因此拆出 `ancestorNumeral`
  两个模式。
  **README 价值**：目前 README 只说"提纲式 / 中文书式"两个词，没解释这解决的是什么真实冲突。
  一句话说清"为什么需要这个开关"比列出选项更能打动真正需要中文书排版的用户。

### 1.5 Backlink 同步：站在已有方案肩膀上做的四处改进

- [§3.12](spec.md#312-backlink-同步m71.0-发布前置)：参考实现是 Header Enhancer 插件的做法
  （逐行配对旧→新标题、`getBacklinksForFile` 反查引用方），本插件在其基础上做了四处改进：
  **原子写入**（`vault.process` 而非非原子的 read+modify）、**保留别名与嵌入**（`|alias`、`!`）、
  **重复标题保守不改**（避免误改）、**改名表非空才进入同步**（日常打字零开销）。
  **README 价值**：README 现有"Rename freely"一节写得好，但没提这是"站在已知最佳实践之上
  又修了几个洞"，而是让读者以为这是从零发明的。可以补一句建立"经过一轮对比调研"的信任感，
  同时不必列出全部四点技术细节（那是 spec 的活）。

### 1.6 两层开关解耦：装了插件 ≠ 到处生效

- [§3.1](spec.md#31-开关命令与生效判定)：**启用插件**（Obsidian 总开关）与**全局自动编号**
  （面板开关）刻意分离——很多用户只想给少数正式文档编号，不愿插件在每次打字时扰动全库。
  frontmatter `true` 可在全局关闭时单独强开某文件；手动命令绕过一切开关（"手动即我要"）。
  **README 价值**：这组"渐进暴露"的开关设计目前分散在"Per-file override""Cleanup commands"
  两节里，读者拼不出这是一套完整的"由粗到细"控制体系。合并陈述会比现在分散更有说服力。

### 1.7 性能承诺：从不批量扫库

- [§4 架构设计](spec.md#4-架构设计)最后一段明确写出边界：插件只在"当前活动文件的
  `editor-change`""手动命令""设置面板改配置后对活动文件即时重排"三个时机触发，且每次只处理
  **当前这一个文件**，从不为"应用了某模板的所有文件"做全库扫描。
  **README 价值**：现有 README 第 52 行只有一句"只解析你正在编辑的文件"，可以补一句
  "模板/规则再多也不会有后台开销"，直接回应大库用户最担心的"这插件会不会拖慢 Obsidian"。

### 1.8 M8 路线图的克制：风险分级，不追求一步到位

- [§5 Roadmap](spec.md#5-roadmap) M8a/M8b 的拆分（本轮此前的规格重整工作）：把"侧栏大纲导航"
  （低风险、只读）与"拖放重排+行内编辑"（高风险、新写入路径）拆成两个独立里程碑，后者要求
  先把核心逻辑设计成不碰 DOM 的纯函数，以便接入现有的约束随机测试。
  **README 价值**：这类内容不适合放正文（用户不关心开发排期），但如果 README 要加"Roadmap"
  或"即将到来"小节，可以用一句话带出"侧栏大纲导航开发中"，不用展开风险分级的内部讨论。

## 2. 竞品对比复核（延续上一轮讨论，补充细节）

| 维度 | 本插件现状 | `number-headings-obsidian`（功能同类） | `obsidian-quiet-outline`（README 风格对标） |
|---|---|---|---|
| 功能定位 | 自动编号 + 模板 + 路径规则 + 白名单 + backlink 同步 | 仅编号，命令触发（非自动防抖） | 大纲导航/跳转，**不做编号**，功能类别不同 |
| README 结构 | 三问题钩子 → 演示 → 功能清单（`###` 平铺）→ 原理 → quick start → 安装 → 命令 → notes | 一句话简介 → 安装 → 使用 → 版本历史（28 版堆到底）→ 致谢 | 12 条功能扁平列表，每条配 GIF，开场即"默认大纲 vs Quiet Outline"对比图 |
| 视觉呈现 | **全篇零图**，代码块演示 backlink 同步 | **全篇零图**，纯文字 | 每个功能一张动图，视觉冲击力强 |
| 内容深度 | 全篇最深——写清楚了取舍与边界（如 backlink 同步限制） | 极浅，靠 changelog 里的 issue 编号堆信任感 | 中等，功能列表说明简短，靠动图补足理解成本 |
| 分层组织 | 无——问题/演示/功能/原理/上手混在一条时间线 | 有基本分层（安装/使用/历史）但内容单薄 | 无——12 条平铺，靠动图弥补扫读成本 |

**结论不变**：本插件在"写了什么"上已经超过两者，短板是"读者能多快找到自己要的那部分"，
以及"抽象功能能不能被一眼看懂"（这正是对方靠动图解决、我方目前只能靠文字硬啃的地方）。
